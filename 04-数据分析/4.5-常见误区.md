# 4.5 常见误区

> 📍 所属：第4章 数据分析 · 层级：误区 · 一句话说明：拆解四个最高频、代价最大的数据分析误区，每个按「表现→为什么会踩→如何避免→自检问题」展开，可直接当评审清单用。

误区之所以是误区，是因为它们**看起来很合理**：数字确实在涨、相关性确实存在、p 值确实小于 0.05。本文四个误区覆盖了指标选择、因果推断、实验执行、数据口径四个环节——恰好对应一次分析从头到尾的四个关口。

---

## 一、用虚荣指标代替健康指标

### 表现

- 汇报永远是累计注册数、累计下载量、总 PV、总曝光——一类"**只会涨、不会跌**"的数字；
- 团队目标定在"注册量翻倍"，于是疯狂投放、注册送礼，注册涨了 3 倍，但 7 日留存从 25% 跌到 8%，付费没有任何变化；
- 数字很好看，但没有人能回答"这个数涨了说明用户得到了什么价值？我们下一步该做什么？"

### 为什么会踩

1. **激励结构**：虚荣指标涨得快、好汇报，向上管理的短期收益明显；健康指标（留存、LTV）见效慢、还可能变差。
2. **认知捷径**：大数字带来天然的成就感，"100 万注册"比"次日留存 31%→34%"更让人兴奋。
3. **缺少北极星**：没有事先约定"什么指标代表真实价值"，汇报时自然挑好看的说。

### 如何避免

- 用两个问题检验每个汇报指标：①它反映用户获得价值了吗？②它变化后我们知道该做什么吗？两问皆否 = 虚荣指标；
- 为团队确立北极星指标 + OSM 体系（见 [4.3-方法论](./4.3-方法论.md)），让所有过程指标向价值指标对齐；
- 给每个"量"指标配一个"质"的搭档一起看：注册量 × 激活率、DAU × 人均核心行为、GMV × 复购率；
- 把累计型指标从周报里请出去，换成对应的流量型/比率型指标（本周新增 + 新增次留，而非累计注册）。

### 自检问题

- [ ] 我汇报的核心指标，有可能在业务恶化时依然上涨吗？（累计数必然如此）
- [ ] 这个指标涨了，我能说出下一步具体动作吗？
- [ ] 我是否给每个规模指标配了质量指标？
- [ ] 如果只允许保留一个指标判断产品生死，我会选现在汇报的这个吗？

## 二、把相关当因果、忽略混杂变量

### 表现

- "用了 XX 功能的用户留存高 20pp，所以我们要强推 XX 功能"——直接把分群相关性当因果结论上手段；
- 强推之后（弹窗、强制引导、任务奖励）行为量确实涨了，留存却纹丝不动，复盘时归咎于"执行不到位"而不是结论本身错了；
- 更隐蔽的版本：忽略混杂变量。"参加会员日活动的用户年消费是未参加者的 3 倍"→ 结论"会员日拉动消费"。实际上是高消费用户本来就更爱参加活动（自选择偏差），活动只是筛选器不是发动机。

### 为什么会踩

1. **叙事本能**：人脑天生爱因果故事，"A 导致 B"比"A 和 B 都由 C 引起"更好讲、更好记；
2. **验证成本不对称**：相关性一条 SQL 就能算，因果验证要设计实验、等两周，deadline 面前人人想抄近道；
3. **结论合意**：当相关性支持自己想推的方案时，人不会主动找茬（确认偏误）。

### 如何避免

- 看到任何相关性结论，强制过"三问"：有没有可能反向因果（Y→X）？有没有共同原因 Z（混杂变量）？有没有自选择（什么样的人才会做 X）？
- 把相关性明确标注为"假设"，用实验验证后才升级为"结论"——记住因果证据阶梯（见 [4.1-第一性原理](./4.1-第一性原理.md)）；
- 做观察性对比时至少控制最明显的混杂：按新老客、活跃度分层后再比（同为高活跃用户中，用没用 XX 功能的留存差异还在吗？分层后差异大幅缩小 = 混杂主导）；
- 汇报措辞自律：用"与……相关""高留存用户更常……"，不用"带来""导致""提升了"，除非有实验证据。

### 自检问题

- [ ] 我这句结论里的"导致/带来/提升"，背后是实验还是分群对比？
- [ ] 做 X 的用户和不做 X 的用户，本身是同一类人吗？
- [ ] 我能想出至少一个第三变量 Z，同时解释 X 和 Y 吗？
- [ ] 如果结论方向对我不利，我还会接受同样强度的证据吗？

## 三、A/B 测试样本不足就下结论、提前偷看

### 表现

- 实验没做样本量计算，"先跑三天看看"，第 2 天 p=0.03 就宣布胜出、火速推全；
- 每天刷实验平台，p 值一旦跌破 0.05 立即截图汇报（peeking）；反之自己不喜欢的方案显著变差时，则"再观察几天"（选择性停止）；
- 推全之后大盘毫无起色，甚至复测不出效果，团队开始怀疑"A/B 测试没用"。

### 为什么会踩

1. **不懂 p 值的波动性**：样本量不足时 p 值剧烈波动，多次查看等于做了多次检验——每天看一眼、看 14 天，即使真实无差异，"至少有一天显著"的概率远超 5%，假阳性率可能被推高到 20% 以上；
2. **业务节奏压力**：版本要上线、活动要开场，"等两周"在排期面前显得矫情；
3. **动机推理**：实验的提出者往往希望自己的方案赢，显著了就想赶紧锁定胜果。

### 如何避免

- **实验开始前**完成并登记三件事：样本量计算（基线、MDE、α、功效四要素，公式见 [4.3-方法论](./4.3-方法论.md)）、预计运行时长（且 ≥7 天覆盖周期效应）、唯一核心指标与判定规则——全部写入 [AB测试记录表](../07-工具与模板库/表格/AB测试记录表.md)，到期才读数；
- 运行期间只监控护栏指标和工程故障，不做"效果结论"；确有提前决策需求，使用支持序贯检验（sequential testing / always-valid p-value）的实验平台，而不是裸看 p 值；
- 建立实验评审制度：推全决策需第二人核查样本量、时长、SRM、分维度一致性；
- 心态设定：**实验的目的是学习，不是给方案盖章**。"不显著"和"负向"都是有价值的产出，避免团队为了"显著"而扭曲流程。

### 自检问题

- [ ] 这次实验开跑前算过样本量吗？MDE 是多少？
- [ ] 实验跑满预定时长/样本量了吗？覆盖完整一周了吗？
- [ ] 我是"到期读数"，还是"看到显著才停"？
- [ ] 分流比例正常吗（SRM 检验）？关键维度下钻后结论一致吗？
- [ ] 如果结果是负向的，我会用同样的标准接受它吗？

## 四、数据口径不统一

### 表现

- 同一个"DAU"，产品报 120 万、数据组报 110 万、老板从投放平台看到 135 万，会议前半小时都在对数而不是做决策；
- 两个团队各自取数：一个把"活跃"定义为"打开 App"，另一个定义为"有核心行为"；一个 GMV 含退款、一个不含；一个按自然日、一个按滚动 24 小时；
- 跨期对比时口径中途变更却无人标注：某月起"新增用户"从设备口径改成账号口径，趋势图上出现一个被当成"业务增长"解读的台阶。

### 为什么会踩

1. **指标名是自然语言**："活跃""新增""收入"人人都懂，人人理解不同——没有强制定义时，每个取数者都会做出自己的默认假设；
2. **数据链路多头**：埋点平台、数仓、业务库、第三方投放后台各有一套数，天然存在统计边界差异（去重方式、时区、丢包）；
3. **缺少治理机制**：没有指标字典、没有认证表，口径知识散落在个人的 SQL 收藏夹里，人一走口径就丢。

### 如何避免

- **建指标字典并强制引用**：核心指标统一登记定义、公式、口径细则、来源表、负责人（模板见 [指标字典模板](../07-工具与模板库/表格/指标字典模板.md)）；报表和分析注明"指标口径见字典 vX.X"；
- **收口取数出口**：核心指标只从数仓认证表（ADS 层，见 [4.2-核心概念](./4.2-核心概念.md)）取，禁止各自从明细层现算大盘数；
- **口径变更走变更管理**：修改定义需通知下游、在趋势图上标注变更点、必要时双口径并行一段时间；
- **交付前过检查清单**：人群范围、时间窗、去重键、是否含退款/测试账号/爬虫，逐项确认（见 [数据口径核对清单](../07-工具与模板库/清单/数据口径核对清单.md)）；
- 遇到数字打架，标准动作是**先对口径再对数**：让双方各自写出取数 SQL 的过滤条件与来源表，九成分歧当场消解。

### 自检问题

- [ ] 我报告中的每个核心指标，能一句话说清它的统计口径吗（人群/时间窗/去重/过滤）？
- [ ] 这个数取自认证表，还是我自己从明细现算的？
- [ ] 和我对比的那组历史数据，口径和现在完全一致吗？
- [ ] 如果别人按我写的口径重新取数，能复现出同一个数字吗？

## 五、把误区清单变成流程

四个误区对应分析链条的四个关口，可直接嵌入团队流程：

| 关口 | 误区 | 拦截机制 |
| --- | --- | --- |
| 定指标 | 虚荣指标 | 北极星 + OSM 评审；量必配质 |
| 下结论 | 相关当因果 | 措辞规范 + 因果证据分级标注 |
| 做实验 | 样本不足/偷看 | 实验登记表 + 到期读数 + 推全双人复核 |
| 出数据 | 口径不统一 | 指标字典 + 认证表收口 + 交付前清单 |

建议每季度用本文四组自检问题对团队的代表性报告做一次"误区体检"。
