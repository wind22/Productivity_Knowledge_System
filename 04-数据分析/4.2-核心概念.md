# 4.2 核心概念

> 📍 所属：第4章 数据分析 · 层级：概念 · 一句话说明：一次讲清指标、统计学、埋点与数据体系的基础词汇，让你和数据团队"说同一种语言"。

概念不清是大多数分析错误的源头：分不清存量与流量会看错趋势，不懂 p 值会误读实验，不了解埋点会提出无法实现的需求。本文按"指标 → 指标体系 → 统计学 → 埋点 → 数据体系"五层展开。

---

## 一、指标基础：四组关键区分

### 1. 绝对值 vs 比率

- **是什么**：绝对值衡量规模（DAU 120 万、GMV 5000 万）；比率衡量效率或质量（转化率 3%、留存率 35%）。
- **为什么重要**：绝对值受盘子大小影响，比率不受；只看其一都会被误导。渠道 A 带来 10 万新增但次留 8%，渠道 B 带来 1 万新增但次留 40%——预算怎么分，取决于你要规模还是要质量。
- **如何应用**：汇报关键结论时**成对呈现**（量 + 率）；比较不同规模对象时优先用比率，但要警惕小样本比率的波动（分母 10 的 50% 不如分母 10000 的 30% 可信）。
- **示例**：某信息流产品两个内容池，池 A 日曝光 1 亿、CTR 1.2%，池 B 日曝光 200 万、CTR 4.5%。正确动作不是"全量切池 B"，而是先验证池 B 的高 CTR 在放量后是否衰减（高比率可能只是小流量下的精选效应）。

### 2. 存量 vs 流量

- **是什么**：存量是某一时点的快照（累计注册用户数、当前会员数）；流量是一段时间内的变化量（本月新增、本月流失）。二者关系：`期末存量 = 期初存量 + 流入 − 流出`。
- **为什么重要**：存量指标"只会涨"（如累计注册数），掩盖健康度；诊断问题必须拆成流入和流出。
- **如何应用**：任何存量指标停滞/下滑时，先做水池拆解：是流入变少，还是流出变多？两者的应对策略完全不同（拉新 vs 挽留）。
- **示例**：某订阅制产品会员数连续 3 个月停在 50 万。拆解发现月新增 6 万、月流失也是 6 万，且流失中 70% 是"首月到期未续费"——问题不在拉新，在新会员首月价值感知，应优先优化 onboarding 而非加投放。

### 3. 先行指标 vs 滞后指标

- **是什么**：滞后指标反映已发生的结果（营收、流失率）；先行指标提前预示结果（激活率、NPS、7 日内核心行为次数）。
- **为什么重要**：滞后指标出问题时往往已无法挽回；管理要盯先行指标，复盘看滞后指标。
- **如何应用**：为每个核心滞后指标找 1~2 个经过相关性验证的先行指标，纳入日常监控并设告警阈值。
- **示例**：SaaS 产品的"年度续费率"是滞后指标（一年才知道结果）。团队验证发现"月活跃席位占比 < 30% 的客户，续费率仅 20%"，于是把"活跃席位占比"作为先行指标，客户成功团队按月干预低活跃客户，次年续费率提升 9pp。

### 4. 虚荣指标 vs 北极星指标

- **是什么**：虚荣指标（Vanity Metrics）好看但不反映真实价值、不指导行动，如累计下载量、页面 PV、注册总数；北极星指标（North Star Metric, NSM）是唯一最能代表"产品向用户交付核心价值"的指标。
- **为什么重要**：团队盯什么指标，就会长成什么样。盯下载量的团队会疯狂买量，盯北极星的团队会优化价值交付。
- **如何应用**：好的北极星指标满足三条——①反映用户获得价值的时刻；②可被团队行动影响；③是营收的先行指标。判别虚荣指标的方法：问"这个数涨了，我们该做什么？"答不上来的就是虚荣指标。
- **示例**：

| 产品 | 虚荣指标 | 北极星指标 |
| --- | --- | --- |
| Airbnb | 网站访问量 | 已完成间夜数（Nights Booked） |
| Spotify | 注册用户数 | 总收听时长 |
| 微信读书 | 下载量 | 周阅读时长/周阅读人数 |
| B2B SaaS | 试用注册数 | 周活跃团队数（WAT） |

## 二、常用指标体系：定义与公式速查表

### 用户规模与活跃

| 指标 | 定义/公式 | 备注 |
| --- | --- | --- |
| DAU / MAU | 日/月活跃用户数（去重） | "活跃"的定义必须写进指标字典：是打开 App，还是完成核心行为？ |
| 粘性比 | `DAU / MAU` | 衡量高频程度；社交/IM 常 >50%，工具类 20% 上下；受拉新影响（大量新客涌入会拉低粘性比） |
| 渗透率 | `功能使用用户数 / 大盘活跃用户数` | 衡量某功能/品类的普及程度 |

### 留存与转化

| 指标 | 定义/公式 | 备注 |
| --- | --- | --- |
| 次日/7日/30日留存率 | `第 N 天回访的用户数 / 当日新增（或活跃）用户数` | 分清"新增留存"与"活跃留存"；分清"第 N 日留存"（精确日）与"N 日内留存"（区间） |
| 转化率 CVR | `完成目标行为用户数 / 进入流程用户数` | 必须声明统计窗口（当日转化 vs 7 日内转化）与去重口径 |

### 商业化

| 指标 | 定义/公式 | 备注 |
| --- | --- | --- |
| GMV | 成交总额（口径需明确：是否含取消/退款/未支付） | 电商常用；GMV ≠ 收入 |
| ARPU | `收入 / 活跃用户数` | 全体活跃用户人均收入 |
| ARPPU | `收入 / 付费用户数` | 付费用户人均收入；`ARPU = ARPPU × 付费率` |
| LTV | 用户生命周期总价值 ≈ `ARPU × 生命周期` 或按留存曲线累加各期收入 | 预估口径差异大，务必注明计算方法与时间窗（如 LTV90） |
| CAC | `获客总成本 / 新增用户数` | 分子是否含人力/内容成本要说清 |
| ROI | `收益 / 成本`（投放场景常用 ROAS = 广告带来的 GMV / 广告花费） | 健康增长的经验线：`LTV / CAC > 3` |

**指标间的骨架关系**（值得背下来）：

```
收入 = DAU × 付费率 × ARPPU
GMV  = 流量 × 转化率 × 客单价
利润 = LTV − CAC （单用户视角）
```

**如何应用**：拿到一个新业务，先用这几条乘法公式把大盘指标拆开，找到最薄弱、最可撬动的因子（详见 [4.3-方法论](./4.3-方法论.md) 的指标拆解）。所有指标定义应沉淀到 [指标字典](../07-工具与模板库/表格/指标字典模板.md)。

## 三、统计学基础：分析师的最小必备集

### 1. 集中趋势：均值、中位数、分位数

- **是什么**：均值 = 总和/个数；中位数 = 排序后中间值；P90 分位数 = 90% 的样本小于该值。
- **为什么**：互联网数据大多是**长尾分布**（少数重度用户/大 R 贡献大头），均值会被极端值拉高，严重偏离"典型用户"。
- **如何应用**：描述"典型用户"用中位数；描述总盘子用均值×人数；监控性能/体验用 P90/P99（如接口耗时 P99）。
- **示例**：某游戏"人均月充值 58 元"，但中位数是 0 元——一半用户根本不充值，均值全靠 0.5% 的大 R 撑起。运营策略应按付费分层设计，而不是面向"人均 58 元"的幻想用户。

### 2. 离散程度：方差与标准差

- **是什么**：方差 `σ² = Σ(xᵢ − μ)² / n`，标准差 σ 是方差开根号，量纲与原数据一致。
- **为什么**：均值相同的两组数据，波动可能天差地别；标准差还是判断"异动是否真的异常"的尺子。
- **如何应用**：监控指标时用"均值 ± 2σ/3σ"设动态告警带，超出才算异常，避免对日常波动大惊小怪。

### 3. 正态分布与长尾分布

- **是什么**：正态分布对称、集中（身高、测量误差）；长尾/幂律分布高度右偏（用户消费、内容播放量、粉丝数）。
- **如何应用**：面对长尾数据——①别用均值做典型描述；②分析前可做 log 变换或分桶；③二八法则先看头部（Top 1% 内容贡献多少 VV？）。
- **示例**：短视频平台 1% 的视频贡献 80% 播放量。评估内容生态健康度时，除了总 VV，更要看"腰部内容占比"这类分布形状指标。

### 4. 置信区间

- **是什么**：用样本估计总体时的不确定性范围。"转化率 3.0%，95% 置信区间 [2.6%, 3.4%]"意思是：这套抽样方法长期使用，约 95% 的情况下区间能罩住真值。样本越大，区间越窄。
- **如何应用**：小样本数据（新渠道前 3 天、灰度 5% 流量）汇报时必须带区间，防止把随机波动当结论。两组数据的置信区间大幅重叠时，别轻易说"A 比 B 好"。

### 5. p 值与统计显著性

- **是什么**：p 值 = 假设两组本无差异（原假设）时，观察到当前或更极端差异的概率。p < 0.05 通常判定"显著"。
- **常见误读**：p 值**不是**"结论正确的概率"，也**不是**"效果大小"。p = 0.001 且提升 0.01% 的结果，统计显著但业务无意义；反之样本不足时 p = 0.2 也不能证明"没效果"，只能说"没测出来"。
- **如何应用**：看实验结果时同时看三样——p 值（是否显著）、效应量（提升幅度）、置信区间（不确定性）。实验记录规范见 [AB测试记录表](../07-工具与模板库/表格/AB测试记录表.md)。

### 6. 辛普森悖论

- **是什么**：分组数据与汇总数据的结论相反——每个子群里 A 都优于 B，合并后 B 反而优于 A。根源是各组**样本量权重不均**。
- **示例**：新旧两个注册流程，新流程在 iOS 转化 50%（vs 旧 45%）、在安卓转化 20%（vs 旧 15%），每端都更好；但新流程恰好被更多投放到低转化的安卓端，汇总转化率反而更低。
- **如何应用**：任何汇总对比，都要用关键维度（平台、渠道、新老客）下钻验证一次；A/B 实验保证分流在各维度上均匀。

## 四、数据类型与埋点

### 1. 三类数据

| 类型 | 内容 | 来源 |
| --- | --- | --- |
| 行为数据 | 谁在什么时间、什么环境做了什么（点击、浏览、播放、搜索） | 埋点上报 |
| 业务数据 | 交易与状态（订单、支付、库存、账户余额） | 业务数据库 |
| 属性数据 | 用户/物品的相对静态特征（注册渠道、城市、会员等级、商品类目） | 注册信息、CRM、标签系统 |

分析的威力来自三类数据的**关联**：行为数据告诉你"过程"，业务数据告诉你"结果"，属性数据让你能"分群"。

### 2. 前端埋点 vs 服务端埋点

| 维度 | 前端埋点（客户端/JS） | 服务端埋点 |
| --- | --- | --- |
| 能采到什么 | 曝光、点击、滑动、停留等交互细节 | 请求、下单、支付等关键事务 |
| 准确性 | 受网络丢包、缓存、版本发布影响，通常有 1%~5% 丢失 | 高，与业务库强一致 |
| 上线成本 | 需发版（App）或改前端代码 | 改服务端代码，随时上线 |
| 典型用途 | 漏斗、路径、体验分析 | 交易统计、对账、核心 KPI |

**实践准则**：钱相关的指标（GMV、支付）以服务端为准；交互体验类指标用前端埋点；两边都有的指标定期对账，差异率超过阈值（如 3%）要排查。

### 3. 事件-属性模型（Event-Property）

- **是什么**：主流埋点范式。每条数据 = 一个**事件（Event）** + 一组**属性（Property）**：

```json
{
  "event": "video_play",
  "user_id": "u_10086",
  "time": "2026-07-05 21:03:11",
  "properties": {
    "video_id": "v_2233",
    "category": "美食",
    "source": "首页推荐",
    "duration_sec": 45,
    "is_finish": false
  }
}
```

- **如何应用**：设计埋点时按"4W1H"检查属性完备性——Who（用户标识）、When（时间）、Where（页面/位置/来源）、What（对象及其特征）、How（方式/结果）。**属性设计比事件数量更重要**：与其埋 50 个孤立事件，不如埋 15 个事件、每个带上完整的 source/位置属性，后者能支持路径与归因分析。
- **示例**：如果"video_play"事件没有 `source` 属性，日后想回答"首页推荐和搜索带来的播放各占多少"就只能重新埋点再等一个月数据。埋点评审时用"未来要回答什么问题"倒推属性清单。

## 五、数据体系分层：一条数据的旅程

### 是什么

从用户点击到老板看到报表，数据经过一条标准流水线：

```
采集（埋点/业务库/日志）
   ↓
数据仓库
   ODS：原始数据贴源层，几乎原样保存
   DWD：明细层，清洗去重、统一字段、还原业务过程（一行=一次行为/一笔订单）
   DWS：汇总层，按主题轻度聚合（用户日粒度行为宽表、商品日销量表）
   ADS：应用层，直接面向报表/业务的结果表（App 日活跃大盘表）
   ↓
BI 报表（固定看板：大盘日报、经营周报）
   ↓
专题分析（临时取数、深度专项：流失分析、定价分析）
```

### 为什么要分层

- **复用与一致**：所有报表从同一 DWD/DWS 出，避免"同一指标各写各的 SQL、结果对不上"；
- **效率**：ADS 预聚合，看板秒开，不必每次扫全量明细；
- **可追溯**：数字有疑问时可逐层回查，定位是采集问题还是加工问题。

### 如何应用（面向分析师/产品/运营）

- 取数时**优先用 DWS/ADS 的认证表**，而不是自己从 ODS 明细现算——又慢又容易口径不一致；
- 发现两张报表数字打架时，第一步是问"分别取自哪一层哪张表、口径是什么"（对照 [数据口径核对清单](../07-工具与模板库/清单/数据口径核对清单.md)）；
- 提数据需求时说清楚要"看板"（长期固定监控 → 走 ADS+BI）还是"专题"（一次性深挖 → 明细层临时分析），两者排期与交付形态不同。

### 互联网示例

某公司运营周会上，增长组说上周新增 12 万，客户端组说 14 万。回查发现：增长组用的 ADS 新增表口径是"首次激活设备"，客户端组自己从 ODS 日志算的是"首次打开事件数"（含重装用户与多次上报）。解决方案不是争论谁对，而是把"新增用户"定义收敛进指标字典、统一从认证表取数。
