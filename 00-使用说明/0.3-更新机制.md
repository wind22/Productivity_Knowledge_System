# 0.3 更新机制

> 📍 所属：第0章 · 本文回答"知识库如何保持活着"：每周复盘沉淀的 SOP、案例入库标准、过期知识的淘汰规则，以及配套的 Git 工作流。

## 一、为什么需要更新机制

知识库的价值 = 内容质量 × 可信度。没有更新机制的库会经历三步死亡：内容过时 → 使用者被坑一次 → 从此不再打开。因此本库把"更新"设计成三个固定动作，而不是"有空再整理"：

| 机制 | 频率 | 解决的问题 |
|------|------|-----------|
| 每周复盘沉淀 | 每周 1 次，约 30-60 分钟 | 新知识持续流入 |
| 案例入库 | 事件驱动（项目结束/看到好案例） | 案例层有真实弹药 |
| 过期知识淘汰 | 每季度 1 次集中复查 + 随手标记 | 存量知识可信 |

## 二、每周复盘沉淀 SOP

### 2.1 是什么、为什么

每周固定一个时间（建议周五下午或周日晚），把本周的输入（读过的文章、开过的会、做过的项目）和踩过的坑，按流程沉淀进库。为什么按周：按天太碎（值得入库的知识没那么多），按月太长（细节已遗忘）。周是"记忆还在 + 素材够量"的平衡点。

### 2.2 复盘五步（可直接照做）

**第 1 步 · 收集（5 分钟）**：翻本周的浏览器收藏、聊天记录中转发给自己的链接、工作文档，列出候选知识点清单。

**第 2 步 · 过滤（10 分钟）**：对每条候选问三个问题，全"是"才继续：
- 三个月后我还可能用到它吗？
- 它对照 [0.1 知识分层模型](0.1-知识分层模型.md) 第四节的入库标准，达标吗？
- 库里已有同类内容吗？（有 → 转为更新既有文档而非新增）

**第 3 步 · 入库（20-30 分钟）**：按（领域, 层级）坐标写入对应文档（规则见 [0.2 知识管理方式](0.2-知识管理方式.md)），只写要点+来源，并补双向链接。

**第 4 步 · 打磨（5 分钟）**：挑一篇本周实际使用过的旧文档，做一层渐进式总结加工（加粗/提炼/补链）。

**第 5 步 · 提交（5 分钟）**：Git 提交并推送，跑一遍坏链检查脚本（见 [0.2](0.2-知识管理方式.md) 第三节）。

**互联网示例**：某周你上线了一个 push 召回实验，效果不及预期。周五复盘时沉淀出三条：① 实验结论"沉默 30 天以上用户 push 打开率不足 1%，召回应前移到沉默 7-14 天窗口"→ 写入 [3.4-关键技巧](../03-运营/3.4-关键技巧.md)；② 踩坑"实验组和对照组的用户活跃度分布不均导致结论差点误判" → 写入 [4.5-常见误区](../04-数据分析/4.5-常见误区.md)；③ 整个实验的完整过程 → 按模板写入 [09-案例库](../09-案例库/README.md)。一次复盘，三层收获。

### 2.3 Git 工作流约定

日常单人维护，用主干直接提交即可，但遵守以下约定：

1. **提交粒度**：一次 commit 只做一类事，一周复盘通常拆成 2-4 个 commit。
2. **Commit message 前缀**（方便日后按类型检索历史）：

| 前缀 | 含义 | 示例 |
|------|------|------|
| `add:` | 新增知识点/文档 | `add: 3.4 增加沉默用户召回时间窗技巧` |
| `polish:` | 渐进式总结打磨、补链接 | `polish: 4.3 漏斗分析小节提炼核心结论` |
| `case:` | 案例入库 | `case: push召回实验复盘` |
| `deprecate:` | 标记过时 | `deprecate: 3.4 微博粉丝头条投放技巧` |
| `remove:` | 删除已淘汰内容 | `remove: 清理2024标记过时的3条技巧` |
| `fix:` | 修坏链、错别字、口径错误 | `fix: 修复0.4指向案例库的坏链` |

3. **大规模重构走分支**：涉及移动文件、改目录结构时开 `refactor/xxx` 分支，改完自查坏链再合并——避免主干出现"链接全断"的中间状态。
4. **每周至少 push 一次**：远端仓库同时是备份。

## 三、案例入库标准（与 09-案例库的衔接）

### 3.1 触发时机

- 自己经手的项目结束（无论成败——失败案例价值往往更高）；
- 读到数据完整、过程可信的外部案例；
- 面试/交流中听到值得记录的一手实践。

### 3.2 入库门槛（四要素齐全才收）

| 要素 | 要求 | 反例 |
|------|------|------|
| 背景 Context | 什么产品、什么阶段、什么问题 | "某大厂"（无阶段无场景） |
| 动作 Action | 具体做了什么，关键决策点在哪 | "优化了推荐算法"（黑箱） |
| 数据 Data | 前后对比数据、口径清晰 | "效果显著提升"（无数字） |
| 结论 Takeaway | 可迁移的经验/教训各一条以上 | 只有故事没有提炼 |

四要素缺一的，先记入个人草稿，等补齐再入库。**警惕二手成功学**：只有结论、数据无法核实的公众号案例，一律不入库。

### 3.3 入库动线

1. 复制 [09-案例库/案例模板](../09-案例库/案例模板.md)，在 `09-案例库/` 下新建文件，命名 `YYYY-领域-一句话主题.md`（如 `2026-运营-push召回时间窗实验.md`）；
2. 按模板填写四要素；
3. **回链**：案例中用到的方法/原理，链接到对应章节；再到对应章节的关联阅读或正文中补一条指向本案例的链接（双链约定见 [0.2](0.2-知识管理方式.md)）；
4. 若案例回答了某个高频场景问题（如"如何提升留存"），到 [INDEX-场景索引](../INDEX-场景索引.md) 对应场景下补一行；
5. 以 `case:` 前缀提交。

## 四、过期知识淘汰

### 4.1 为什么必须淘汰

一条被证伪的技巧比十条缺失的知识更危险——它会以"库里写着呢"的权威姿态误导决策。各层半衰期不同（见 [0.1](0.1-知识分层模型.md)），淘汰重点是技巧层和案例层。

### 4.2 标记方式（随手做）

任何时候发现某条知识可疑或已失效，**不要直接删**，先标记：

1. 在该文档标题下方（导读引用块之前）加一行醒目标记：

```markdown
> ⚠️ 已过时（2026-07 标记）：微信生态外链规则已变化，本文玩法大部分失效。替代阅读：[3.3-方法论](../03-运营/3.3-方法论.md)
```

2. 若只是文档内某一小节过时，则在该小节标题后加 `（⚠️ 已过时）`，并在小节开头一句话说明原因；
3. 以 `deprecate:` 前缀提交。

标记必须包含三要素：**标记时间、过时原因、替代去处**（若有）。

### 4.3 季度集中复查

每季度最后一周的复盘时间延长为一次"大扫除"：

1. 列出候选：`git log --since="1 year ago" --name-only --pretty=format: | sort | uniq -c | sort -n` 找出一年内几乎没被动过的文档，加上所有 `⚠️ 已过时` 标记；
2. 逐条判决：
   - **仍有效** → 顺手做一层打磨，留下 `polish:` 提交记录；
   - **失效但有史料价值**（如经典案例）→ 保留 ⚠️ 标记，正文不动；
   - **失效且无价值** → 删除正文，`remove:` 提交（Git 历史里永远找得回，不需要"回收站"目录）；
3. 检查坏链，更新受影响的 [INDEX-场景索引](../INDEX-场景索引.md) 条目。

**互联网示例**：2023 年前很多运营库里都有"微信裂变海报玩法"技巧。规则收紧后，正确做法不是假装它不存在，而是加上 `⚠️ 已过时（2023-05 标记）：企业微信外链规则调整，诱导分享玩法已被限流`，并指向仍然有效的上位方法（社交传播的底层逻辑在 [1.4-用户心理学基础](../01-底层思维/1.4-用户心理学基础.md)）。读者由此同时获得两个信息：这条路不通了，以及为什么当年通。

## 五、更新机制自检清单

每月自查一次，三个以上"否"说明机制已失灵，需要降低标准而不是弃更：

- [ ] 过去四周是否至少完成 3 次每周复盘？
- [ ] 本月是否至少新增或更新 1 个案例？
- [ ] 是否存在"知道过时但没标记"的内容？
- [ ] 坏链检查是否通过？
- [ ] Commit 历史是否能看出 `add/polish/case` 的均衡分布（只有 add 说明库在膨胀而非沉淀）？
