# 0.2 知识管理方式

> 📍 所属：第0章 · 本文说明本库采用的三种知识管理方法（MECE 分类、双链笔记、渐进式总结），以及它们在"纯 Markdown + Git"环境下的具体落地做法。

## 一、三种方法各管什么

| 方法 | 解决的问题 | 作用阶段 |
|------|-----------|----------|
| MECE 分类 | 知识放哪里，不重复、不遗漏 | 入库时 |
| 双链笔记 | 知识之间如何关联，形成网络 | 入库后 |
| 渐进式总结 | 知识如何越用越精炼 | 每次重读时 |

三者是流水线关系：**MECE 决定位置 → 双链建立连接 → 渐进式总结持续压缩**。

## 二、MECE 分类

### 2.1 是什么、为什么

MECE（Mutually Exclusive, Collectively Exhaustive）即"相互独立、完全穷尽"：分类之间不重叠，加起来不漏项。

为什么重要：知识库最常见的死法是"同一条知识存了三处、每处都不完整"。写的人随手放，找的人到处翻，最终没人再信任这个库。MECE 让每条知识有且只有一个"户口所在地"。

### 2.2 本库的 MECE 结构

本库在两个维度上做了 MECE 切分：

**横向——按领域切**（02~05 章）：产品 / 运营 / 数据分析 / 商业分析，四个领域对应互联网公司四类核心岗位职能，边界清晰。

**纵向——按层级切**（每章内部）：原理 / 概念 / 方法 / 技巧（含误区）五层，见 [0.1 知识分层模型](0.1-知识分层模型.md)。

两个维度交叉，任何一条知识的坐标 = （领域，层级），例如"漏斗分析步骤" = （数据分析，方法层）→ 落在 [4.3-方法论](../04-数据分析/4.3-方法论.md)。

**处理"切不干净"的部分**：现实中总有跨界知识，本库用三个专门章节兜住，避免破坏主结构的 MECE：

- 跨领域的底层规律 → [01-底层思维](../01-底层思维/README.md)
- 天然跨领域的主题（增长、商业化、策略、AI）→ [06-交叉领域](../06-交叉领域/README.md)
- 具体案例（一个案例常涉及多领域）→ [09-案例库](../09-案例库/README.md)

### 2.3 落地做法（Markdown + Git）

1. **一条知识只有一个正文位置**。其他地方需要它时，用相对路径链接过去，绝不复制粘贴正文。
2. **入库前先定坐标**：问两个问题——"它属于哪个领域？它回答 Why/What/How/Hack/Case 中的哪个？"两问答完，文件位置唯一确定。
3. **拿不准坐标的，宁可先放宽**：跨两个领域的放 06 章；层级模糊的先按方法层收录，复盘时再调整（见 [0.3 更新机制](0.3-更新机制.md)）。
4. **目录结构即分类法**：不新建目录、不建"其他/杂项"文件夹。想建"杂项"说明分类没想清楚，回到第 2 步。

**互联网示例**：一篇讲"如何用 A/B 测试验证新功能"的笔记，涉及产品（功能决策）和数据分析（实验方法）。按规则拆分：实验设计流程写入 [4.3-方法论](../04-数据分析/4.3-方法论.md)（正文唯一位置），[2.3-方法论](../02-产品/2.3-方法论.md) 的"功能验证"小节只写一句话摘要并链接过去。半年后无论从产品还是数据入口进来，看到的都是同一份最新版本。

## 三、双链笔记

### 3.1 是什么、为什么

双链（Bi-directional Links）指笔记之间互相引用，形成网状结构而非树状孤岛。树状目录回答"它属于哪"，链接网络回答"它和谁有关"——后者才是知识产生复利的方式：解决真实问题时，你需要的从来不是单条知识，而是一条"原理→方法→案例"的知识链。

### 3.2 落地做法（Markdown + Git）

纯 Markdown 没有 Obsidian 那样的自动反链，本库用三条约定实现"穷人版双链"：

1. **正向链接**：正文中提到其他文档的概念时，直接用相对路径链接，如 `[北极星指标](../02-产品/2.2-核心概念.md)`。规则：一个概念在一篇文档中首次出现时必须带链接。
2. **反向链接**：每篇文档末尾固定设 `## 关联阅读` 小节，列出"谁应该和我一起被阅读"。当你在 A 文档中链接了 B，顺手到 B 的关联阅读里补一条指回 A 的链接（成对维护）。
3. **链接检查**：定期用命令扫描坏链，作为每周复盘的固定动作：

```bash
# 提取所有 .md 中的相对链接并检查目标文件是否存在
grep -RhoE '\]\((\.\.?/[^)#]+\.md)' --include='*.md' . | \
  sed 's/](//' | sort -u | while read f; do
    [ -f "$f" ] || echo "坏链: $f"
  done
```

**互联网示例**：你在 [09-案例库](../09-案例库/README.md) 写了一个"某短视频产品用签到任务拉升次留"的案例。写完后建三条链：案例正文链向 [3.3-方法论](../03-运营/3.3-方法论.md) 的用户激励体系（它用了什么方法）、链向 [1.4-用户心理学基础](../01-底层思维/1.4-用户心理学基础.md) 的损失厌恶（它为什么有效）；再回到这两篇的关联阅读里补上指向案例的反链。三个月后你研究"激励体系"时，顺着反链一步就能找到这个案例，而不是靠搜索碰运气。

## 四、渐进式总结（Progressive Summarization）

### 4.1 是什么、为什么

渐进式总结由 Tiago Forte 提出：不要在收藏时一次性做完加工，而是**每次重读、每次使用时加工一层**——第一层存原文，第二层加粗重点，第三层高亮精华，第四层写自己的摘要，第五层输出成自己的作品。

为什么有效：加工发生在"真正用到的时候"，注意力天然聚焦在有价值的部分；从没被重读过的内容自然停留在低层，成为淘汰候选（见 [0.3 更新机制](0.3-更新机制.md)）——加工程度本身就是知识价值的投票器。

### 4.2 落地做法（Markdown + Git）

Markdown 没有高亮笔，本库把五层映射为如下操作：

| 层 | 原版做法 | 本库做法 |
|----|----------|----------|
| L1 摘录 | 保存原文 | 有价值的原始材料记入个人草稿或直接以要点形式写进文档小节，注明来源 |
| L2 加粗 | 加粗关键句 | 重读时对关键结论使用 `**加粗**` |
| L3 高亮 | 荧光笔 | 把小节中最核心的一句提炼为该小节开头的一句话（或引用块） |
| L4 摘要 | 写执行摘要 | 在文档导读（📍 引用块）和章 README 中用一两句话概括本文 |
| L5 再创作 | 输出作品 | 提炼为 [07-工具与模板库](../07-工具与模板库/README.md) 的模板/清单，或沉淀为跨文档的原理层结论 |

配套约定：

1. **允许文档长期"半成品"**。刚入库的小节可以只有 L1 级的要点罗列，不追求一次写成完美——完美主义是知识库停更的头号原因。
2. **重读即加工**：每次因为解决实际问题而打开一篇文档，离开前至少做一层加工（加粗一句、补一个链接、改一处过时表述），用 Git 提交，commit message 前缀 `polish:`。
3. **用 Git 历史观察热度**：`git log --oneline -- 路径` 一眼看出哪些文档被反复打磨（高价值）、哪些一年没动过（复查候选）。

**互联网示例**：你读到一篇讲"用户激励体系设计"的长文。第一周只把 6 个要点记进 [3.3-方法论](../03-运营/3.3-方法论.md)（L1）；一个月后做签到功能时重读，把"激励要兑现在用户核心动线上"加粗（L2）并提到小节开头（L3）；季度末发现这条被引用了三次，于是把它扩写成 [07-工具与模板库](../07-工具与模板库/README.md) 里的"激励体系设计清单"（L5）。整个过程没有一次性的大工程，知识却越用越锋利。

## 五、三种方法的协作全流程

以"收藏了一篇好文章"为例，完整走一遍：

```
1. 拆解：文章里有哪些独立知识点？各属于哪个（领域, 层级）坐标？   ← MECE
2. 入库：按坐标写入对应文档，只写要点+来源，不求完美            ← 渐进式总结 L1
3. 连接：给新内容加正向链接，到被链文档补反向链接               ← 双链
4. 提交：git commit（见 0.3 的工作流约定）
5. 复用：日后每次用到，加工一层、补一条链                      ← 渐进式总结 L2-L5
```

## 六、知识条目的质检标准：三问

一个知识点写没写到位，用三个问题验收——**答不全三问的条目是"资料"，答全了才是"知识"**：

| 问题 | 检验什么 | 不合格的样子 |
|------|----------|--------------|
| 什么时候用？ | 适用场景与边界写清了吗 | 只有定义，不知道什么问题该想起它 |
| 怎么用？ | 有步骤/公式/表格可照做吗 | "要重视用户价值"式的正确废话 |
| 怎么判断用对了？ | 有产出标准或自查点吗 | 照做了也不知道做得对不对 |

三问不要求每个条目都写成固定小节（概念层条目通常前两问就够），但**方法层条目建议补齐三个字段**，写作时对照：

- **输入**：用这个方法之前需要准备什么信息/数据（如漏斗分析的输入是"定义清晰的环节事件+口径一致的数据"）；
- **输出**：用完应该产出什么（一份文档？一个判断？一张表？）；
- **自查点**：1-3 条"用对了没有"的检验项（如 MECE 拆解的自查点是"各分支加总能还原总量吗"）。

每次渐进式总结加工旧文档时（见第四节），顺手用三问过一遍，不达标的补上——这比一次性全库整改现实得多。
