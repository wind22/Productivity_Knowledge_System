# 0.4 索引方式

> 📍 所属：第0章 · 本文解释本库的"双入口"检索设计：按领域索引（学习时用）与按问题场景索引（干活时用），以及两个入口如何维护。

## 一、为什么需要两个入口

使用知识库有两种截然不同的场景：

| 场景 | 典型问题 | 检索特征 |
|------|----------|----------|
| 系统学习 | "我想补齐数据分析的知识体系" | 自上而下，按学科结构浏览 |
| 解决问题 | "老板问为什么这个月留存跌了，怎么查？" | 自下而上，从问题直奔答案，5 分钟内要找到 |

单一的目录树只服务第一种场景。真实工作中 80% 的检索是第二种——带着具体问题来，而问题从不按目录结构长：一个"留存下降"的问题横跨数据分析（怎么定位原因）、产品（功能是否有问题）、运营（是否活动退坡）。因此本库设计双入口：

```
入口一：按领域索引（根 README.md）
        领域 → 章 → 五层文档          适合：系统学习、建立框架

入口二：按场景索引（INDEX-场景索引.md）
        问题场景 → 直达相关小节/案例   适合：带着问题来、快速定位
```

两个入口指向同一批文档——**知识只存一份，入口可以有多个**（这也是 [0.2](0.2-知识管理方式.md) MECE 原则"正文唯一、链接多处"的体现）。

## 二、入口一：按领域索引（根 README）

### 2.1 结构

根目录 `README.md` 是领域入口，按"章 → 文档"两级组织，与目录树一致：

- [00-使用说明](README.md)：本库怎么用（元知识）
- [01-底层思维](../01-底层思维/README.md)：跨领域的原理层
- [02-产品](../02-产品/README.md) / [03-运营](../03-运营/README.md) / [04-数据分析](../04-数据分析/README.md) / [05-商业分析](../05-商业分析/README.md)：四大领域，各按五层组织
- [06-交叉领域](../06-交叉领域/README.md)：增长、商业化、策略、AI 能力
- [07-工具与模板库](../07-工具与模板库/README.md)：拿来即用的模板/表格/清单
- [08-学习路径与资源](../08-学习路径与资源/README.md)：学习路径与书单
- [09-案例库](../09-案例库/README.md)：所有案例

### 2.2 如何使用

- **零基础入门某领域**：先读该章 README 了解结构，再按 `x.1 → x.2 → x.3` 顺序读（Why → What → How），配合 [8.1-分阶段学习路径](../08-学习路径与资源/8.1-分阶段学习路径.md)；
- **有经验者查漏**：直接扫该章 `x.4-关键技巧` 和 `x.5-常见误区`，对照自己的实践找盲区。

**示例**：一位后端转岗产品的同学，从根 README 进入 [02-产品](../02-产品/README.md)，两周内按 2.1→2.5 顺序通读，建立框架；之后遇到具体问题就不再走这个入口，改走场景索引。

## 三、入口二：按问题场景索引（INDEX-场景索引.md）

### 3.1 是什么

根目录的 [INDEX-场景索引.md](../INDEX-场景索引.md) 是一张"问题 → 知识"的映射表：左边是工作中真实会被问到的问题（用使用者的语言，而不是学科术语），右边是分散在各章的相关文档链接，按建议阅读顺序排列。

### 3.2 场景条目的标准格式

每个场景一个小节，包含三部分：场景问题（标题）、知识链（按"先方法后原理"排序的链接列表）、相关案例。示例：

```markdown
### 场景：如何提升留存？

**先看方法（How）**
1. [4.3-方法论](../04-数据分析/4.3-方法论.md) —— 留存分析：先定位是哪个群体、哪个环节在流失
2. [2.3-方法论](../02-产品/2.3-方法论.md) —— 上手体验与核心价值路径优化
3. [3.3-方法论](../03-运营/3.3-方法论.md) —— 用户分层运营与激励体系

**再补原理（Why）**
- [1.4-用户心理学基础](../01-底层思维/1.4-用户心理学基础.md) —— 习惯回路：用户为什么会留下来
- [2.1-第一性原理](../02-产品/2.1-第一性原理.md) —— 留存的本质是持续兑现用户价值

**相关案例（Case）**
- [09-案例库](../09-案例库/README.md) 中标注"留存"主题的案例
```

注意两个设计选择：

1. **顺序是"方法在前、原理在后"**——带着问题来的人先要解法，读完解法再决定是否深挖原理。这与学习入口（原理在前）刻意相反。
2. **链接精确到文档并注明看什么**——每条链接后面用一句话说明"这篇解决你问题的哪一段"，避免点开后还要自己再找。

### 3.3 场景从哪里来

场景清单不是一次性设计出来的，而是从真实检索行为中长出来的，来源有三：

- 自己每次带着问题查库，如果"想了一下才知道该去哪章"，就说明这是一个该建索引的场景；
- 工作中被高频问到的问题（"这个功能该不该做""这次活动 ROI 怎么算"）；
- 每周复盘（见 [0.3 更新机制](0.3-更新机制.md)）时新增的内容，若明显服务某个场景，顺手补索引。

典型的初始场景集（供参考）：如何提升留存 / 如何做一次活动复盘 / 如何判断一个功能该不该做 / 数据异动如何排查 / 如何写竞品分析 / 如何设计 A/B 实验 / 如何做商业化变现选型。

## 四、双入口的维护规则

入口是索引，索引会腐烂。维护规则如下：

| 时机 | 动作 |
|------|------|
| 新增文档 | 更新所属章的 README；若明显服务某场景，同步补 INDEX 条目 |
| 新增案例 | 在 INDEX 相关场景的"相关案例"下补一行（见 [0.3](0.3-更新机制.md) 案例入库动线第 4 步） |
| 标记 ⚠️ 已过时 | 检查 INDEX 中指向它的条目，改指替代文档或加同样标记 |
| 季度复查 | 走查 INDEX 全部场景：链接是否可达、顺序是否仍合理、有没有该建未建的高频场景 |

两条红线：

1. **INDEX 只放链接和一句话说明，不放正文**——否则它会变成第三个内容存放地，破坏"正文唯一"原则；
2. **场景数量宁少勿滥**——只收录真实发生过至少 2 次的检索需求；30 个精准场景远比 100 个臆想场景有用。

## 五、检索决策树（怎么选入口）

```
我现在要找东西
 ├─ 带着具体问题？（"怎么办/为什么跌/该不该做"）
 │    └─ 走 INDEX-场景索引 → 没有对应场景？
 │         ├─ 能判断领域 → 去该章 x.3 方法论 / x.4 技巧
 │         ├─ 判断不了 → 全库搜索：grep -Ri "关键词" --include='*.md' .
 │         └─ 找到答案后：回头在 INDEX 补上这个场景（下次 5 秒直达）
 ├─ 想系统学习某领域？
 │    └─ 走根 README → 章 README → 按 x.1→x.5 顺序读
 └─ 想找模板/清单直接干活？
      └─ 直达 07-工具与模板库/工具清单.md
```

**互联网示例**：周一早会老板问"上周北极星指标为什么掉了 5%"。你打开 [INDEX-场景索引](../INDEX-场景索引.md) 的"数据异动如何排查"场景，第一条链到 [4.3-方法论](../04-数据分析/4.3-方法论.md) 的异动分析流程（维度拆解 → 排除口径/日历因素 → 定位群体），第三条链到一个类似的历史案例。10 分钟后你带着排查框架和初步假设进会议室——这就是场景入口存在的意义：把"想起来该看什么"的时间压缩到接近零。
