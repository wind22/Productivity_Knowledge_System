# 8.3 案例库建设指南

> 📍 所属：第8章 学习路径与资源 · 本文说明如何按"背景→动作→结果→可迁移经验"四段格式把实战经验沉淀为案例，以及入库标准、打标签方法和与每周复盘机制的衔接。

## 〇、为什么要有案例库

原理和方法论章节（02~06 章）回答"一般情况下该怎么做"，但知识只有嵌在具体情境里才好记、好信、好用。案例库是本知识库的"证据层"：

- **对自己**：把经验从"当时的感觉"固化为"可复查的记录"，对抗遗忘和记忆美化；
- **对团队**：新人看十个案例，比听十次口头传授吸收得快；
- **对知识库**：方法论章节的每条方法，理想状态下都应有案例佐证或反驳——案例是校验知识的实验数据。

案例统一存放在 [09-案例库](../09-案例库/README.md)，写作直接复制 [案例模板](../09-案例库/案例模板.md)。

## 一、什么样的案例值得入库

不是所有工作都值得写案例。入库前用下面的标准过滤：

### 1.1 三条入库标准（满足任意一条即可）

1. **有反直觉的结论**：结果与做之前的预期明显不符（比预期好或差都算）。符合预期的常规操作写进去只是流水账。
2. **有可复用的决策过程**：面临过真实的方案取舍（A 还是 B），且取舍的理由未来还会用到。
3. **有代价高昂的教训**：踩了坑，且这个坑不在现有的误区章节（x.5）里，或在但当时没能识别。

### 1.2 三类不值得入库

- **纯执行记录**：没有决策、没有意外，"按计划做完了"——放周报即可；
- **无法验证结果的**：动作做了但结果数据拿不到/说不清——先补数据，补不齐就放弃；
- **不可言说原因的成功**："也不知道为什么就涨了"——可以先记一条待办，等归因清楚再入库，切忌编造归因。

一个经验值：**每月 1-2 个高质量案例，胜过每周 3 个流水账**。案例库的敌人是稀释。

## 二、四段格式：每段写什么

四段结构"背景→动作→结果→可迁移经验"是刻意设计的因果链：情境（为什么是这个问题）→ 决策（为什么是这个动作）→ 证据（动作产生了什么）→ 抽象（哪些能带走）。四段完整字段见 [案例模板](../09-案例库/案例模板.md)，这里讲每段的写作要点。

### 2.1 背景：让半年后的读者能"代入当时"

写三件事：**业务上下文**（产品阶段、用户规模、团队情况——半年后连你自己都会忘）、**面临的问题**（用数据描述，如"次日留存从 41% 跌到 33%"，而非"留存不太好"）、**约束条件**（时间、人力、预算、技术、合规——约束决定了为什么最优解不可行，漏写约束的案例最容易被错误迁移）。

自检问题：一个不了解这个项目的同事读完背景，能否说出"如果是我，我大概会考虑哪几个方向"？

### 2.2 动作：写决策，不只写操作

三个必写点：

- **做了什么**：按时间线或逻辑线列出关键动作，具体到可复现（"优惠券面额从 20 减 5 改为 20 减 8，推送时间从 10 点改为 21 点"，而非"优化了优惠券策略"）；
- **为什么这么做**：每个关键动作背后的假设是什么、依据是什么（数据？用户访谈？竞品参考？拍脑袋也要如实写）；
- **备选方案**：当时考虑过但放弃的方案及放弃理由。这是四段中最常被偷懒省略、却最有长期价值的部分——它记录的是你的决策模型。

### 2.3 结果：量化 + 与预期对比

- **量化数据**：核心指标的前后对比，注明统计口径与观察时长；能做显著性检验的（如 A/B 测试）写检验结果；
- **与预期对比**：做之前预期什么，实际差多少。这一栏逼你在行动前就有显式预期——没有预期就无所谓验证；
- **诚实记录副作用**：主指标涨了但护栏指标跌了？写出来。只报喜的案例库会系统性误导后来者。

### 2.4 可迁移经验：从"这件事"到"这类事"

- **可复用结论**：用"当…时，…往往…"的句式抽象（如"当活动激励与用户既有目标一致时，小额激励即可撬动行为"），而不是复述事实；
- **适用边界**：明确写"什么情况下这条结论不成立"。没有边界的经验是危险品——案例的价值一半在结论，一半在边界；
- **关联知识点**：链接到本库对应的原理/方法章节（如链到 [4.3 数据分析方法论](../04-数据分析/4.3-方法论.md) 的实验设计小节）。若案例修正了某章节的说法，按 [0.2 知识管理方式](../00-使用说明/0.2-知识管理方式.md) 的双链约定，回到该章节的"关联阅读"补一条指回案例的链接。

## 三、如何打标签便于检索

纯 Markdown 环境下，标签写在案例元信息区（模板已预留），检索靠 `grep`。约定如下：

### 3.1 三维标签体系

| 维度 | 说明 | 取值示例 |
| --- | --- | --- |
| 领域 | 案例主要归属（与 09 章子目录一致，单选） | `产品` `运营` `数据分析` `商业分析` `增长` |
| 主题 | 涉及的具体话题（可多选，2-4 个为宜） | `留存` `转化率` `AB测试` `定价` `冷启动` `推送` `活动` `漏斗` |
| 类型 | 案例的性质（单选） | `成功` `失败` `反直觉` `决策复盘` |

书写格式（一行，`#` 前缀便于精确匹配）：

```
标签：#运营 #留存 #推送 #反直觉
```

### 3.2 使用规则

1. 领域标签必须且只有一个，与文件所在子目录一致（目录管"放哪"，标签管"怎么搜"）；
2. 主题标签优先复用已有词——打标签前先 `grep -rh "^标签：" 09-案例库/ | tr ' ' '\n' | sort | uniq -c` 看看现有词表，避免"AB测试/A-B测试/ab实验"三种写法并存；
3. 检索示例：找所有失败的留存类案例——`grep -rl "#留存" 09-案例库/ | xargs grep -l "#失败"`。

完整标签词表与维护规则见 [09-案例库 README](../09-案例库/README.md)。

## 四、与每周复盘机制的衔接

案例库不靠热情维护，靠机制维护。它挂接在第 00 章的每周复盘流程（见 [0.2 知识管理方式](../00-使用说明/0.2-知识管理方式.md) 及 00 章的更新机制）上，每周复盘时固定执行三步：

1. **筛（5 分钟）**：过一遍本周的工作，用第一节的三条入库标准筛选。多数周筛出 0-1 个，属正常；
2. **记（2 分钟或 30 分钟）**：达到入库标准但数据未齐的，在案例文件里先建骨架（填好背景与动作，结果留空并标注"待补：X月X日回填数据"）；数据已齐的，复制 [案例模板](../09-案例库/案例模板.md) 完整写完；
3. **连（5 分钟）**：为新案例补双链——案例的"关联知识点"链向章节，对应章节的"关联阅读"链回案例；同时清一遍库里的"待补"标记，数据已到期的当场回填。

另有两个低频动作：**每季度**横向重读本季案例，把重复出现的结论上升为方法论章节的正式内容（案例是知识的孵化器，不是终点站）；**每半年**清理一次——结论已被吸收进章节正文、且无独立参考价值的案例，可在文首标注"已归档：结论已并入 [某章节]"。

## 五、常见问题

- **写一个案例要多久？** 熟练后 30-45 分钟。超过 1 小时通常是背景写太长或试图一次写完美——先完成，复盘时再迭代（渐进式总结，见 [0.2](../00-使用说明/0.2-知识管理方式.md)）。
- **失败案例会不会显得自己很差？** 案例库的记分方式和绩效相反：一个诚实的失败案例比十个包装过的成功案例值钱。团队使用时建议明确"案例免责"共识。
- **该写多细？** 以"半年后的自己 / 一个新同事能否据此复现决策过程"为标准。机密数据可脱敏（写相对变化不写绝对值），但不要脱敏到失去信息量。
