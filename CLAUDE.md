# CLAUDE.md

本仓库是一个纯 Markdown 的**互联网综合知识库**（产品 × 运营 × 数据分析 × 商业分析），不是代码项目。对本仓库的任何修改都必须遵守以下约定。

## 仓库结构

- `00-使用说明/` ~ `09-案例库/`：按章节组织，`02`~`05` 四个领域章均为五件套：`X.1-第一性原理` → `X.2-核心概念` → `X.3-方法论` → `X.4-关键技巧` → `X.5-常见误区`，每章有 README 导航。
- `07-工具与模板库/`：下分 `模板/`、`表格/`、`清单/` 三个子目录，全部要求"拿来即用"（字段带填写说明与示例）。
- 根目录双入口：`README.md`（按领域）+ `INDEX-场景索引.md`（按问题场景）。
- `scripts/check_links.py`：质量检查脚本；`.github/workflows/link-check.yml` 在 CI 中运行它。

## 文档格式规范（新增/修改任何 .md 都必须遵守）

1. **YAML front matter**（文件最前面，CI 强制校验）：

   ```yaml
   ---
   title: 方法论            # 不含编号的标题
   chapter: 03-运营         # 所在章目录名
   section: "3.3"           # 章内编号，README/模板等无编号文件可省略
   layer: 方法              # 原理|概念|方法|技巧|误区|专题|元知识|导航|模板|表格|清单|工具|资源|案例|索引
   domain: 运营             # 通用|产品|运营|数据分析|商业分析|交叉|工具|资源|案例
   tags: [AARRR, 积分体系]  # 2-6 个检索关键词
   updated: 2026-07-05      # 最后实质性修改日期
   ---
   ```

2. **一级标题**后紧跟 **📍 导读引用块**：`> 📍 所属：第X章 · 一句话说明本文解决什么问题`。
3. 正文每个知识点独立小节，包含：概念解释（是什么/为什么）→ 如何应用（How）→ 互联网行业示例。
4. 文末设 `## 关联阅读`，用**相对路径**链接相关文档。
5. 内容过时不删除：在导读下方标注 `⚠️ 已过时`，注明原因与替代结论（见 `00-使用说明/0.3-更新机制.md`）。

## 概念权威页规则（防口径漂移）

- 跨章概念（RFM、LTV/CAC、A/B 测试等）**只在权威页深入展开**，其他位置一句话概述 + 链接。
- 权威页登记表：`00-使用说明/0.5-概念权威页对照表.md`。**修订跨章概念前先查表；新增跨章概念先登记再动笔。**
- 更新概念只改权威页，引用处的概述除非本身有错否则不动。

## 新增文档时的挂载动作（三处，缺一不可）

1. 所在章的 `README.md` 导航表加一行；
2. 根 `README.md` 对应章节表加一行；
3. 若回答某个业务问题，在 `INDEX-场景索引.md` 挂载入口（格式：问题 → 知识点链接 → 模板/清单链接，每场景 3-5 条链接）。

## 提交前检查

- 运行 `python3 scripts/check_links.py`，必须全部通过（断链 + front matter 校验）。
- 模板内**有意**指向不存在路径的占位链接，须加入脚本内 `ALLOWED_BROKEN` 白名单并注释原因。

## Git 约定

- 提交信息前缀（见 `00-使用说明/0.3-更新机制.md`）：`add:`（新增知识）、`polish:`（修订/精炼）、`case:`(案例入库)、`deprecate:`（标记过时）、`remove:`（淘汰删除）、`fix:`（断链/格式修复）。
- 案例文件命名：`09-案例库/<领域>/YYYYMM-案例主题.md`，格式按 `09-案例库/案例模板.md` 四段式（背景→动作→结果→可迁移经验）。

## 写作风格

- 简体中文，GitHub 风格 Markdown；示例优先用中国互联网产品。
- 内容要求"可直接指导工作"：给公式、给表格、给步骤，不写空洞套话。
- 语言平实，避免堆砌形容词；一个小节只讲一件事。
