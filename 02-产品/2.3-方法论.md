# 2.3 方法论

> 📍 所属：第2章 产品 · 层级：方法 · 本文提供从需求收集到项目落地的全流程可执行方法，接到任务时按图索骥

方法论覆盖产品工作的完整链路：**收集需求 → 分析需求 → 理解用户 → 规划路线 → 设计产品 → 写 PRD → 管理项目 → 研究竞品**。每节给出步骤、要点和示例。

## 一、需求收集与管理

### 1.1 四大收集渠道

| 渠道 | 适用场景 | 优点 | 陷阱 |
| --- | --- | --- | --- |
| 用户访谈 | 探索"为什么"，挖深层动机 | 深度高、能追问 | 样本小、易被善辩用户带偏 |
| 问卷调查 | 验证假设、量化分布 | 样本大、可统计 | 只能验证已知问题，问不出未知 |
| 数据洞察 | 发现"用户实际在做什么" | 客观、规模大 | 只见行为不见动机（What 无 Why） |
| 竞品分析 | 补盲区、找基准 | 成本低、快 | 容易变成抄袭，抄到别人的错误 |

**组合拳原则**：数据发现异常（What）→ 访谈探究原因（Why）→ 问卷验证规模（How Many）。单一渠道得出的结论都不可靠。

**访谈要点**：问过去的事实，不问未来的意愿（"你上次遇到这个问题是什么时候？当时怎么解决的？"而不是"如果有 X 功能你会用吗？"——用户对未来的承诺几乎不作数）。

### 1.2 需求池管理

所有需求进入统一需求池（表格或 Tapd/飞书多维表格），最小字段集：

```
需求ID | 一句话描述（用户+场景+问题） | 来源 | 提出人 | 影响用户量估算 | 价值假设 | 状态（待评估/排期中/已上线/已拒绝+理由）
```

三条纪律：

1. **有进有出**：每周固定时间清理，超过 3 个月无人认领的需求归档。
2. **拒绝也要留痕**：写明拒绝理由，避免同一需求被反复提出、反复讨论。
3. **进池不等于会做**：需求池是漏斗的入口，不是承诺清单——这句话要明确告知所有需求提出方。

## 二、需求分析：KANO 模型与四象限

### 2.1 KANO 模型

按"具备时的满意度"和"不具备时的不满度"把功能分为五类：

| 类型 | 具备时 | 不具备时 | 策略 | 示例（微信） |
| --- | --- | --- | --- | --- |
| 基本型（必备） | 无感 | 极度不满 | 必须做，做到无短板 | 消息能收到、不丢 |
| 期望型（越多越好） | 满意 | 不满 | 主战场，持续投入 | 消息同步速度、语音质量 |
| 兴奋型（魅力） | 惊喜 | 无感 | 挑 1-2 个做差异化 | 早期的"摇一摇"、拍一拍 |
| 无差异型 | 无感 | 无感 | 不做 | 开屏动画的第 5 种样式 |
| 反向型 | 不满 | 满意 | 坚决不做 | 强制已读回执（对 C 端用户） |

**实操**：对每个功能设计正反两问（"如果有 X 你觉得如何？/如果没有 X 你觉得如何？"，选项从"很喜欢"到"很不喜欢"），投放给 30+ 目标用户，按 KANO 对照表归类。注意：兴奋型会随时间退化为基本型（当年的"在线支付"是兴奋型，今天是基本型），每年应重测。

### 2.2 优先级四象限

以**价值（影响用户量 × 需求强度 × 商业收益）**为纵轴、**成本（人日 + 技术风险）**为横轴：

| | 低成本 | 高成本 |
| --- | --- | --- |
| **高价值** | 立即做（Quick Win） | 规划做（战略项目，拆分成里程碑） |
| **低价值** | 顺手做（塞进迭代缝隙） | 不做（明确拒绝并留痕） |

进阶可用 RICE 打分量化：`RICE = (Reach 触达 × Impact 影响 × Confidence 信心) / Effort 成本`，适合需求量大、争议多的团队，用分数减少"拍脑袋"争论。

## 三、用户研究：Persona、Journey Map、JTBD 访谈法

### 3.1 Persona（用户画像）

**是什么**：从真实调研数据中抽象出的典型用户原型，让团队讨论时"心中有同一个人"。

**怎么做**：定性访谈 10-20 人 → 按行为变量（而非人口属性）聚类 → 每类提炼为一个具象人物（姓名、场景、目标、痛点、常用产品）→ 全团队可见。

**要点**：Persona 必须基于数据而非想象（想象出来的叫"自嗨画像"）；数量控制在 2-4 个，并指定 1 个首要 Persona——功能冲突时以首要 Persona 为准。

**示例**：K12 教育 App 至少要区分"焦虑型家长（付费决策者）"与"被安排的学生（使用者）"两个 Persona，两者的目标几乎对立（管控 vs 自由），产品的家长端/学生端设计因此完全不同。

### 3.2 Journey Map（用户旅程图）

**是什么**：把用户完成任务的全过程按时间轴展开，标注每个触点的行为、想法、情绪，找出体验低谷（=机会点）。

**结构**：横轴为阶段（认知→考虑→使用→留存→推荐），每列填四行——用户行为 / 接触点 / 情绪曲线（-2 到 +2 打分）/ 痛点与机会。

**示例**：外卖旅程中"下单后等待"是情绪最低谷（焦虑、无掌控感），美团把机会点做成了骑手实时地图 + 精确到分钟的送达预估——没有加快配送，却显著改善了体验（呼应 [2.1](./2.1-第一性原理.md) 的"体验=感知−预期"）。

### 3.3 JTBD 访谈法

围绕"用户最近一次真实的雇佣/解雇行为"做深访，还原决策全过程：

1. 定位事件："你最近一次开始用/放弃用 X 是什么时候？"
2. 回溯时间线：第一次想到要解决这个问题是什么时候？触发事件是什么？
3. 挖掘四种力量：**推力**（旧方案哪里不行）、**拉力**（新方案吸引在哪）、**焦虑**（换用新方案怕什么）、**惯性**（旧习惯为何难改）。
4. 结论落成 Job Statement："当我（场景），我想要（动机），以便（结果）。"

**应用**：增长停滞时，用"焦虑+惯性"清单找转化阻力（比拉新更便宜的增长往往在这里）。

## 四、产品规划：Roadmap、版本节奏、OKR 对齐

### 4.1 Roadmap（路线图）

**是什么**：以季度/半年为颗粒度的产品演进蓝图，回答"我们按什么顺序逼近产品愿景"。

**怎么做**：愿景 → 拆成 3-5 个战略主题（如"供给丰富度""履约体验""商业化"）→ 每个主题按季度排里程碑 → 与研发、运营、市场对齐资源。

**要点**：Roadmap 承诺的是**方向和主题**，不是精确到天的功能清单（越远越模糊是正常的：本季度到功能级，下季度到主题级，明年只到方向级）。

### 4.2 版本节奏

| 节奏 | 适用 | 说明 |
| --- | --- | --- |
| 双周迭代 | 多数 App/Web 产品 | 固定发版日，需求赶不上就等下班车，用节奏倒逼拆分 |
| 大版本 + 小步快跑 | 平台级产品 | 季度大版本（营销节点）+ 双周小版本（修复与AB实验） |
| 持续交付 | 服务端/B端 SaaS | 功能开关控制发布，随时可上（配合灰度，见 [2.4](./2.4-关键技巧.md)） |

### 4.3 OKR 对齐

产品规划必须能向上对齐公司 OKR：每个 Roadmap 主题标注支撑哪条 O / KR；对不上任何 OKR 的项目要么砍掉，要么说明它是"探索性投资"并控制在总资源的 10-20% 以内。反向同样成立：如果公司 KR 是"次月留存提升 5pp"，而 Roadmap 里没有任何留存向项目，规划就是失职。

## 五、产品设计：信息架构、尼尔森十大可用性原则、设计规范

### 5.1 信息架构（IA）

**是什么**：内容与功能的组织、命名和导航体系，决定"用户能不能找到"。

**方法**：卡片分类法（让目标用户对功能卡片自由分组归类，据此设计导航）；树测试（给用户任务，看能否在纯文字目录树中找到正确路径）。

**原则**：广度与深度平衡（移动端一级导航 ≤ 5 个，任务路径 ≤ 3 层）；按用户心智分类而非按公司组织架构分类（用户不关心这是哪个部门的功能）。

### 5.2 尼尔森十大可用性原则（评审走查清单）

1. 系统状态可见（上传要有进度条）
2. 贴近真实世界（用"购物车"不用"暂存队列"）
3. 用户可控可撤销（误删可恢复，微信消息 2 分钟内可撤回）
4. 一致性（同一操作全站同一说法与位置）
5. 防错优先于报错（不可用日期直接置灰，而非选完再弹错）
6. 识别优于回忆（展示最近搜索，别让用户回忆上次搜了什么）
7. 灵活高效（给高频用户快捷方式：手势、常购清单）
8. 极简（每屏一个主要任务）
9. 帮用户识别、诊断、恢复错误（报错要说人话并给出口："支付失败，银行卡余额不足，试试换张卡？"）
10. 帮助文档（放在触手可及处，而非唯一依赖）

**用法**：交互稿评审时逐条走查，每条给 0-4 分严重度，≥3 分的问题必须改完才能进开发。

### 5.3 设计规范（Design System）

统一的色彩、字号、组件库（按钮、表单、弹窗、空状态）。价值：设计与前端效率提升 30%+，体验一致性有保障。中小团队不必自研，基于 Ant Design（B端）或各平台官方规范（iOS HIG / Material Design）裁剪即可。

## 六、PRD 撰写

PRD 的本质是**对齐工具**：让设计、研发、测试、运营对"做什么、为什么做、做到什么程度算完成"达成一致。完整模板见 [PRD 模板](../07-工具与模板库/模板/PRD模板.md)，核心五块：

| 模块 | 写什么 | 常见缺陷 |
| --- | --- | --- |
| 背景与目标 | 需求来源、数据/用户证据、要改善的指标及目标值 | 只写"老板要的"；目标不可量化 |
| 用户故事 | 作为（角色），我想要（功能），以便（价值）；按优先级排列 | 写成功能清单，丢失"以便"（价值缺失无从取舍） |
| 功能逻辑 | 流程图 + 状态机 + 边界条件（为空/超限/断网/并发/权限） | 只写正常流程，异常分支留给开发"自由发挥" |
| 埋点需求 | 事件名、触发时机、属性字段、对应验证的指标 | 上线后才想起要数据（见 [2.4 埋点前置](./2.4-关键技巧.md)） |
| 验收标准 | 可判真伪的条件清单（Given-When-Then 格式） | 写"体验流畅"这类无法验收的形容词 |

**验收标准示例**（Given-When-Then）：

```
Given 用户购物车中有失效商品
When 用户点击"结算"
Then 弹出失效商品清单，支持一键移除，移除后自动重新计算金额并进入结算页
```

## 七、项目管理：敏捷/Scrum、瀑布、双钻模型

### 7.1 敏捷 / Scrum

**核心思想**：小步快跑，用短周期交付换取快速反馈，拥抱变化。

Scrum 最小实践集：固定 Sprint（2 周）→ Sprint 计划会（从 Backlog 领任务并估点）→ 每日站会（昨天/今天/阻塞，15 分钟）→ 评审会（演示可用的产品增量）→ 回顾会（流程改进）。产品经理承担 PO 角色：维护 Backlog 排序、随时解答需求疑问、验收。

**适用**：需求不确定、需要快速试错的 C 端产品（国内互联网主流）。

### 7.2 瀑布

需求 → 设计 → 开发 → 测试 → 发布严格分阶段，前一阶段冻结后才进入下一阶段。**适用**：需求确定、变更代价极高的场景——金融核心系统、与硬件/外部机构联调的项目（如支付机构与银联的对接排期）。互联网团队常用混合式：大框架瀑布定里程碑，内部敏捷迭代。

### 7.3 双钻模型（Double Diamond）

设计思维的宏观流程，两次"发散-收敛"：

```
问题域钻石              方案域钻石
发现(发散) → 定义(收敛) → 发展(发散) → 交付(收敛)
广泛调研     锁定真问题    多方案探索    聚焦最优方案落地
```

**价值**：强制团队"先把问题搞对，再把方案做对"。多数失败项目死于跳过第一颗钻石——拿着老板给的方案直接进开发。用法：立项评审设两道关卡，第一道只审"问题定义对不对"，第二道才审方案。

## 八、竞品分析框架：定位 → 功能 → 数据 → 策略推演

四步递进，从"看见"到"预判"（配套模板见 [竞品分析模板](../07-工具与模板库/模板/竞品分析模板.md)）：

| 步骤 | 分析什么 | 信息来源 |
| --- | --- | --- |
| 1. 定位 | 目标用户、核心场景、商业模式、slogan 演变 | 官网、应用商店描述、融资新闻、招股书 |
| 2. 功能 | 功能清单对比、核心流程走查（截图存档）、版本更新日志时间线 | 亲自深度使用 ≥ 2 周、七麦/点点数据的版本记录 |
| 3. 数据 | DAU/MAU、下载量趋势、留存估算、收入规模 | QuestMobile、七麦、财报、电话会纪要 |
| 4. 策略推演 | 它最近的动作指向什么意图？下一步最可能做什么？对我们的威胁与机会？ | 综合 1-3 交叉推理，输出行动建议 |

**关键纪律**：

- 分析必须落到**行动建议**（我们跟进/差异化/忽略，理由是什么），没有结论的竞品分析是资料汇编。
- 版本更新日志是金矿：把竞品近 12 个月的更新日志按主题归类，其资源投入方向一目了然。
- 警惕"功能对齐陷阱"：竞品有 ≠ 我们该有，对方的战略、用户结构和你不同（呼应 [2.5 常见误区](./2.5-常见误区.md)）。

**示例**：2020 年前后分析"多多买菜 vs 美团优选"，仅对比功能几乎一样；但从定位层看，拼多多是防守（保卫下沉市场基本盘、复用农产品供应链），美团是进攻（把即时配送网络的密度优势延伸到次日自提）——同样的功能，完全不同的战略含义，推演出的补贴持续性和区域优先级判断也不同。

## 九、全流程串联：一个需求的标准旅程

```
渠道收集 → 进需求池 → KANO/四象限定级 → （必要时）用户研究验证
→ 进 Roadmap/版本排期（对齐 OKR）→ 交互设计（IA+尼尔森走查）
→ PRD 评审 → Scrum 迭代开发 → 灰度发布 → 数据验证 → 复盘回池
```

每个环节都可能"杀死"需求——这是正常的，漏斗的价值就在于过滤（参见 [2.1](./2.1-第一性原理.md)：产品经理的本质是约束下的价值判断）。
