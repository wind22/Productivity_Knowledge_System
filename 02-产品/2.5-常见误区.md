# 2.5 常见误区

> 📍 所属：第2章 产品 · 层级：误区 · 本文拆解四个最高频的产品大坑，每个按「表现 → 为什么会踩 → 如何避免 → 自检问题」展开，供立项前与复盘时对照自检

误区之所以是误区，是因为它们在当下看起来**都是对的**：功能在增加、方案很酷、声音大的需求确实紧急、设计得更完善"总没错"。本文的价值是提供事前识别的特征和自检清单。

## 误区一：把功能当产品、把方案当需求（"更快的马"）

### 表现

- 团队讨论永远围绕"做什么功能"，从不讨论"解决谁的什么问题"；季度汇报的成绩单是"上线了 47 个功能"，而不是任何指标的变化。
- 用户/业务方说"我要一个 X 按钮"，就原样写进 PRD 排期开发——把用户口中的**方案**直接当成了**需求**。
- 典型台词："竞品有这个功能，我们也得有""老板说要做个社区"。

亨利·福特那句被反复引用的话是最好的注脚："如果你问顾客想要什么，他们会说——一匹更快的马。"用户能准确感知**问题**（去镇上太慢了），但只能在自己的认知范围内提出**方案**（马）。汽车不在他的词汇表里。

### 为什么会踩

1. **方案是具体的，需求是抽象的**：讨论按钮和页面有画面感、有进度感，追问"背后的问题"显得务虚又得罪人，大脑天然选省力的路（参见 [1.2 通用心智模型](../01-底层思维/1.2-通用心智模型.md) 中的认知吝啬鬼）。
2. **交付压力**：排期紧张时，"翻译需求"这一步最先被省略——照做最快，照做也最安全（"是你要的我才做的"）。
3. **提出者位高权重**：老板/大客户给的往往直接是方案，反问需要勇气和技巧。

### 如何避免

- **强制翻译**：任何进入需求池的条目必须还原成「用户 + 场景 + 问题」三要素（见 [2.3 方法论](./2.3-方法论.md)），原始方案只作为附注保留。
- **五问法**：对每个方案连问"这是为了解决什么问题？"直到触底。示例：客服部门要求"增加机器人自动回复"→ 为什么？→"人工回复不过来"→ 为什么回不过来？→"70% 都在问'我的退款到哪了'"→ 真需求是**退款进度透明化**。在订单页加一条退款进度条，可能比做整套机器人便宜十倍、效果好十倍。
- **对高层方案做"目标确认"而非"正面拒绝"**："您希望这个功能带来什么变化？我拿数据核一下现状，下周给您两三个达成该目标的方案对比。"——既尊重了意图，又拿回了方案权。
- 汇报口径改造：功能清单 → 指标变化。逼自己为每个功能回答"它改变了什么"。

### 自检问题

- [ ] 我能不用提及任何功能名词，只用「谁、在什么场景、遇到什么问题」把这个需求讲清楚吗？
- [ ] 如果禁止使用当前方案，我还能想出至少两种解决同一问题的替代方案吗？（想不出=我根本没理解问题）
- [ ] 这个"需求"的原始提出人描述的是他的困难，还是他想象中的解法？
- [ ] 上线后我用什么指标判断"问题被解决了"，而不是"功能被做完了"？

## 误区二：自嗨式创新（未验证 PMF 就大规模投入）

### 表现

- 团队对创意极度兴奋，但兴奋的证据全部来自内部："我们都觉得很酷""老板拍板了""发布会反响热烈"。
- 还没有一批真实用户持续留下来，就开始：招人扩编、买量投放、开发二期三期功能、谈商务合作。
- 数据侧的典型症状：下载量/注册量很好看（投放买来的），但**留存曲线一路下滑到接近零**、次月几乎无人回访；团队用"新增"掩盖"留存"。
- 典型台词："先把规模做起来，体验后面再优化""这个赛道窗口期很短，等不了验证"。

### 为什么会踩

1. **宜家效应与沉没成本**：人会高估自己亲手做出来的东西的价值；投入越多越不愿承认方向错了（见 [1.4 用户心理学基础](../01-底层思维/1.4-用户心理学基础.md)）。
2. **幸存者偏差的叙事**：媒体讲的都是"押注成功"的故事，没人报道一万个死于同样打法的项目。
3. **确认偏误的调研**："你觉得这个主意怎么样？"式的提问只会收获礼貌性赞美——朋友、同事、被拉来的用户都不想扫兴。
4. **组织激励错位**：大项目、大预算、大团队 = 个人影响力，"小步验证"在晋升答辩上不好看。

### 如何避免

- **PMF 前只花"学费"，不花"扩张费"**：给验证阶段设硬性资源上限（如 3 人 3 个月），验证通过的标准**事先写下**——Sean Ellis 40% 测试、同期群留存曲线走平、自然增长占比等（判断标准详见 [2.2 核心概念](./2.2-核心概念.md)）。事先写下是关键，事后定标准永远能自圆其说。
- **用行为验证，不用口头验证**：预约页愿意留手机号吗？愿意付 9.9 元订金吗？愿意把它转发给同事吗？真金白银和真实行为才算票。
- **主动找否定证据**：调研时问"你上次遇到这个问题是什么时候？当时怎么解决的？"（过去的事实），而不是"如果有 X 你会用吗？"（未来的客气话）。
- **设置"熔断复盘"**：投放前约定——若第 4 周次周留存 < X%，无条件暂停扩张、回到产品迭代。让规则代替当时的意气做决定。

### 案例

共享经济泡沫期（2016-2017）的共享雨伞、共享篮球、共享充电宝一齐涌现，多数玩家在毫无留存验证的情况下铺量投放设备。事后只有共享充电宝活了下来——它恰好命中"手机没电"这一高频、高痛感、场景明确的真需求；而共享篮球的场景（到了球场却没带球、且愿意扫码付费）真实频次极低。同样的商业模式模板，PMF 有无之差就是生死之差。反面对照：微信读书上线后长期保持小团队慢迭代，直到"社交读书+无限卡"模式在留存数据上被验证后才加大推广。

### 自检问题

- [ ] 支持这个项目的证据中，来自"真实用户持续使用行为"的有几条？来自"团队内部感觉"的有几条？
- [ ] 我们的同期群留存曲线走平了吗？还是靠新增在填流失的窟窿？
- [ ] 验证通过/失败的标准是在投入之前白纸黑字写下的吗？
- [ ] 如果明天停掉所有投放和补贴，用户还会来吗？我怎么知道？

## 误区三：需求排期只看声音大小

### 表现

- 排期表实际上是"权力表"：老板提的排最前，嗓门大、催得勤的业务方其次，沉默的普通用户的需求永远垫底。
- 同一个需求，换个更高级别的人来提，优先级立刻变化；评审会开成吵架会，谁能扛住压力谁的需求上。
- 客服每天转来的高频用户抱怨（影响几十万人）排不上，某大客户销售的定制要求（影响一家）插队直上。
- 版本规划毫无主线：这个迭代是各方施压结果的大杂烩，做完之后没有任何一个指标被显著改善。

### 为什么会踩

1. **可得性偏差**：天天在耳边响的需求显得更重要；沉默用户的需求没有"音量"，只存在于数据里（见 [1.2 通用心智模型](../01-底层思维/1.2-通用心智模型.md)）。
2. **产品经理缺少裁决框架**：没有公开的优先级标准，就只能被最强的外力推着走——不是 PM 想谄媚，是他手里没有能对抗压力的武器。
3. **短期激励**：安抚眼前的人立竿见影，服务沉默的大多数收益滞后。
4. **组织文化**：如果公司里"向上交付"比"向用户交付"更受奖励，排期自然向权力倾斜。

### 如何避免

- **建立公开的排序规则**：用 RICE 或价值/成本四象限（见 [2.3 方法论](./2.3-方法论.md)）给所有需求打分，打分过程和结果对所有需求方透明。规则的意义不是绝对精确，而是**让插队的成本从"吼一嗓子"上升为"公开论证"**。
- **给沉默用户发声道具**：把客服工单 TOP10、应用商店差评聚类、NPS 追问文本做成月度报告抄送所有需求方——让数据替不在会场的人说话（呼应 [2.4 关键技巧](./2.4-关键技巧.md) 的"用数据和用户声音代替个人观点"）。
- **老板需求同样过框架**：不是拒绝，而是透明化代价："可以做，它会替换掉本迭代的 A 需求（预计影响留存 KR 约 1pp），您确认这个交换吗？"多数时候老板要的是被认真对待，而不是无条件服从。
- **保留配额制**：每个迭代预留固定比例（如 20%）给"数据驱动的用户体验需求"，不接受任何挤占——用制度保护无声者。

### 自检问题

- [ ] 本迭代需求列表里，有几个来自数据/用户研究，有几个来自"谁催得急"？
- [ ] 如果把所有需求的提出人姓名遮住，按同一套标准重新打分，排序会大变吗？（会=排期被权力扭曲了）
- [ ] 最近三个月，有没有一个"没人催但数据支持"的需求被成功排上并上线？
- [ ] 我拒绝声音大的需求时，给出的是框架和数据，还是只能硬扛？

## 误区四：过度设计

### 表现

- 第一版就要"完整"：用户系统要有七种角色权限，运营后台要支持所有可配置项，"以后肯定用得上"。
- 为 1% 的边缘场景把主流程搞复杂：为防止极小概率的作弊，给 99% 的正常用户增加三步验证。
- 追求技术/设计上的优雅超过解决问题：自研组件库、预留十倍扩展性，而产品还没验证 PMF。
- 功能上线后数据：核心功能使用率 90%，精心设计的高级选项使用率 0.3%，但它们占用了 40% 的开发时间。
- 典型台词："万一以后要支持呢？""既然做了就一步到位。"

### 为什么会踩

1. **损失厌恶与不确定性焦虑**："以后返工"被想象成巨大损失，于是用"现在多做"来买安心——但实际上多数"以后"永远不会来（YAGNI：You Aren't Gonna Need It）。
2. **专业人士的诅咒**：设计师/工程师/PM 的专业审美驱使他们把东西做"完善"，而用户只想完成任务。做得越多越显得专业、越有成就感。
3. **把想象中的用户当真实用户**：边缘场景来自会议室头脑风暴而非数据，脑补的场景没有频次约束，可以无限繁殖。
4. **缺少"成本"意识**：每个"顺便加上"的选项都有隐性成本——开发、测试、维护、以及**用户的认知负担**（选项每多一个，主路径就模糊一分）。

### 如何避免

- **用 MVP 纪律对抗**：第一版只做"验证核心价值闭环"所需的最小集合（见 [2.2 核心概念](./2.2-核心概念.md)）。判断标准：砍掉这个东西，核心闭环还成立吗？成立就砍。
- **给边缘场景定"数据门槛"**：任何为边缘场景增加的复杂度，先回答"该场景实际发生频率是多少？数据在哪？"没有数据的，用人工兜底（客服处理）代替产品化——等真实频次撑起来再做。
- **KANO 无差异型排雷**：疑似"锦上添花"的功能先做 KANO 问卷（见 [2.3 方法论](./2.3-方法论.md)），无差异型直接出局。
- **复盘时算"功能使用率账"**：每季度盘点功能使用率，长期 < 1% 的功能列入下线候选。下线功能和上线功能一样是产出。
- 参考反面基准：飞书/钉钉的早期版本、微信的"小而美"路线——微信语音通话入口藏了很多年，直到通话需求被数据确证为高频才逐步前置。

### 案例

对照组是最好的老师：12306 早期改版曾提供极其完备的车次筛选条件（十几个维度的组合查询），但绝大多数用户的任务只是"从 A 到 B，尽量有票"——后来的候补购票功能用"你要去哪+没票帮你排队"一个动作解决了主任务，使用率远超那一堆高级筛选。同期，航旅纵横、高铁管家等第三方 App 正是靠"把主任务做到一步直达"切走了大量用户。复杂不是能力的证明，找准并压缩主路径才是。

### 自检问题

- [ ] 砍掉这个功能/选项/配置项，核心价值闭环还成立吗？（成立=现在不该做）
- [ ] 这个"边缘场景"有真实数据支撑频次吗，还是来自会议室的"万一"？
- [ ] 我是在为用户的任务设计，还是在为自己的专业成就感设计？
- [ ] 第一版发布日期如果强制提前一半，我会砍掉什么？——那些东西为什么现在在范围内？

## 四个误区的共同根源与总自检

四个误区可以归结为同一个失败模式：**用"做事的感觉"替代了"价值的验证"**。

| 误区 | 被替代的验证 | 一句话解药 |
| --- | --- | --- |
| 把方案当需求 | 问题是否真实存在 | 先翻译成「用户+场景+问题」 |
| 自嗨式创新 | 市场是否真的要（PMF） | 留存曲线走平之前不谈扩张 |
| 排期看声音 | 价值排序是否成立 | 公开的打分框架 + 让数据替沉默者发声 |
| 过度设计 | 复杂度是否被频次支撑 | MVP 纪律 + 边缘场景数据门槛 |

**立项前的终极三问**（建议加入 [功能上线前检查清单](../07-工具与模板库/清单/功能上线前检查清单.md) 的前置环节）：

1. 问题真实吗？——证据是用户的过去行为，不是任何人的想象。
2. 值得现在做吗？——对齐目标、频次×强度、机会成本都算过了。
3. 做到多小就够了？——先验证闭环，再谈完善。
