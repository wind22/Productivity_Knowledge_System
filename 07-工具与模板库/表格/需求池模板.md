# 需求池模板

> 使用说明：需求池是团队所有待办需求的唯一入口，建议每周固定时间（如周一）评审一次：新需求补全字段并打分，存量需求更新状态。
>
> 使用规则：
> 1. 任何来源的需求先入池再讨论，不入池的需求不排期——避免"谁嗓门大做谁的"；
> 2. RICE 打分由产品负责人主持，E（人力）由研发估；分数是排序参考而非唯一依据，战略型需求可显式加权，但要写明理由；
> 3. 「拒绝」也是一种状态，拒绝的需求保留在池中并写明理由，避免同一需求反复被提。

---

## 一、字段说明

| 字段 | 填写要求 |
|---|---|
| 需求ID | 唯一编号，建议 REQ-年份-流水号；PRD、上线记录均引用此 ID 追溯 |
| 来源 | 用户反馈 / 业务方（写具体团队）/ 数据洞察 / 老板 / 竞品跟进 / 技术债——来源影响验证方式 |
| 需求描述 | 一句话讲清"要什么"，描述问题而非直接给方案（"用户找不到历史订单"优于"加个搜索框"） |
| 用户场景 | 谁、在什么情况下、遇到什么问题；附证据（反馈条数、数据截图链接） |
| 价值评估 | 定性描述：支撑什么目标（北极星/收入/合规/体验），量级预估 |
| R（Reach 触达） | 单位时间内受影响的用户数（如每季度触达人数），填数字 |
| I（Impact 影响） | 对单个用户的影响强度：3=巨大，2=高，1=中，0.5=低，0.25=极小 |
| C（Confidence 信心） | 对 R 和 I 估计的信心：100%=有数据实证，80%=有部分数据，50%=纯直觉 |
| E（Effort 人力） | 预估投入，单位「人月」，含产品/设计/研发/测试 |
| RICE 总分 | = R × I × C ÷ E，分数越高优先级越高 |
| 状态 | 待评估 / 已评估待排期 / 已排期 / 开发中 / 已上线 / 已拒绝 / 挂起 |
| 排期 | 目标迭代版本或时间；已拒绝/挂起的写理由 |

## 三、RICE 打分速查

| 打分项 | 取值指引 |
|---|---|
| Impact | 3 = 核心指标显著变化 / 2 = 明显提升 / 1 = 中等 / 0.5 = 轻微 / 0.25 = 几乎无感 |
| Confidence | 100% = A/B 或强数据支撑 / 80% = 有相关数据 / 50% = 直觉判断；低于 50% 说明该需求应先做调研而非排期 |
| Effort | 取整到 0.5 人月；超过 6 人月的需求建议拆分后再入池打分 |

**注意事项**

- RICE 天然低估「体验型小需求」和「技术债」，建议每个迭代固定预留 15-20% 容量给这两类，不与功能需求抢分数；
- 合规类需求不参与打分，直接标记「合规-必做」并按截止时间排期；
- 每季度清理一次：连续两个季度未排期且信心无提升的需求，转为「挂起」或「已拒绝」。

## 四、需求评审会议流程（每周，建议 45 分钟）

1. **新需求过池（20 min）**：提出人 2 分钟内讲清「问题、场景、证据」；字段不全的退回补全，不占用会议讨论；
2. **打分（15 min）**：当场给 R/I/C/E 打分。争议超过 3 分钟的需求，标记「待调研」并指定调研负责人，下周再议——会上吵不出数据；
3. **状态刷新（10 min）**：已排期/开发中的需求核对进度；上线满 30 天的需求核对「价值评估」是否兑现，未兑现的记入复盘素材。

## 五、常见反模式对照

| 反模式 | 症状 | 纠正方法 |
|---|---|---|
| 需求池变垃圾场 | 几百条需求无人维护，谁也不信这张表 | 严格执行季度清理；池内保持 <60 条活跃需求 |
| 用方案伪装需求 | 描述里全是"加个按钮/弹窗" | 强制改写为问题描述，方案讨论留给 PRD 阶段 |
| 分数崇拜 | 完全按 RICE 分数机械排期 | 分数是排序参考；战略/合规/技术债走独立通道并留容量 |
| 老板需求跳过打分 | 高层需求直接插队且无场景描述 | 照样入池打分；分数低不等于不做，但差距要摆在桌面上供决策 |
| 只进不出 | 「已拒绝」状态从未被使用 | 每次评审至少明确拒绝 1-2 条，拒绝理由留档 |

## 六、字段裁剪建议

- 小团队（<5 人）：可去掉 RICE 四列，只保留「价值评估 + 状态 + 排期」，靠对话对齐；
- 平台型/B 端团队：建议增加「客户名称/合同关联」「承诺交付日」两列，商业承诺型需求单独标色；
- 数据条件成熟后：R 尽量取自真实埋点数据而非拍脑袋，C 相应上调，分数可信度随之提高。
