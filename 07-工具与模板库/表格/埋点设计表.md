# 埋点设计表

> 使用说明：本表随 PRD 一起评审——**没有埋点表的 PRD 不进开发**（埋点前置的理由见 [2.4 关键技巧](../../02-产品/2.4-关键技巧.md)）。事件-属性模型的概念见 [4.2 核心概念](../../04-数据分析/4.2-核心概念.md)（权威页）。
>
> 三条纪律：
> 1. **每个事件必须回答"服务哪个指标/分析"**——答不上来的埋点不加（埋点不是越多越好，垃圾事件会永久污染数仓）；
> 2. **触发时机写到无歧义**——"用户看到商品"不合格，"商品卡片露出面积 ≥50% 且停留 ≥500ms"才合格；
> 3. **涉钱事件以服务端为准**——客户端埋点有丢报（行业常见丢失率 3-10%），支付、下单、发奖类事件必须服务端埋点或双端对账。

---

## 一、命名与设计规范

| 规范 | 约定 | 示例 / 反例 |
|---|---|---|
| 事件命名 | `对象_动作` 小写下划线；全局唯一；动作用一般现在时 | ✅ `checkout_click`、`course_card_impression`；❌ `点击结算`、`ClickCheckoutBtnNew2` |
| 属性命名 | 小写下划线；跨事件同义属性必须同名同类型 | `course_id` 在所有事件中都是 string，不允许一处叫 `courseId` |
| 公共属性 | 所有事件自动携带，不在每个事件里重复设计 | user_id、device_id、os、app_version、channel、时间戳 |
| 枚举值 | 属性取值提前枚举并写死，禁止自由文本 | `pay_method`: alipay / wechat / applepay，新增值需登记本表 |
| 版本管理 | 事件结构变更只加属性不改语义；语义变了就新建事件并废弃旧的 | 旧事件标注"⚠️ 2026-07 起废弃，改用 xxx" |

## 二、事件设计表

<!-- 一行一个事件。示例为电商结算链路，替换为你的业务 -->

| 事件名 | 中文名 | 触发时机（精确描述） | 上报端 | 事件属性（名:类型:取值） | 服务的指标/分析 | 优先级 | 负责人 | 验收状态 |
|---|---|---|---|---|---|---|---|---|
| `checkout_click` | 点击结算 | 购物车页点击"结算"按钮时（不论后续校验是否通过） | 客户端 | cart_amount:float; sku_count:int; coupon_selected:bool | 购物车→结算转化率 | P0 | 张三 | ✅ 已验数 |

## 三、公共属性表

| 属性名 | 类型 | 说明 | 采集方 |
|---|---|---|---|
| user_id | string | 登录用户 ID，未登录为空 | SDK |
| device_id | string | 设备唯一标识，匿名分析主键 | SDK |
| app_version | string | 语义化版本号，用于新老版本对比 | SDK |
| channel | string | 安装渠道 / UTM 来源 | SDK |
| session_id | string | 会话标识（30 分钟无操作切新会话） | SDK |

## 四、三类高频坑（设计时对照）

1. **曝光口径歧义**：曝光是"进入视口"还是"停留够时长"？列表快速滑动算不算？——口径不同 CTR 能差 3 倍。规范里写死露出比例 + 停留时长 + 去重规则，且全产品统一。
2. **客户端与服务端打架**：客户端 `pay_success` 与服务端订单数对不上是常态（丢报、重试、回调时序）。规则：**过程看客户端、结果看服务端**；两端差异 >5% 触发排查（见 [数据口径核对清单](../清单/数据口径核对清单.md)）。
3. **改版打断可比性**：改事件语义不改名，新旧数据混在同一事件里，历史趋势全部作废。语义变更 = 新事件 + 旧事件标注废弃日期。

## 五、上线验收清单

- [ ] 每个 P0 事件在测试环境实际触发一次，后台能查到且属性完整（逐条截图留档）
- [ ] 触发时机与本表描述逐字核对（尤其曝光类、去重规则）
- [ ] 枚举属性的每个取值都触发验证过，无未登记的野值
- [ ] 涉钱事件完成客户端-服务端对账，差异 <5%
- [ ] 事件已登记进团队埋点管理平台/本表，负责人与服务指标可追溯
- [ ] 上线 24 小时后回查数据量级是否符合预期（量级差 10 倍通常是触发时机埋错了）
