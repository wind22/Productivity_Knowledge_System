# 商业化风控清单

> 使用说明：凡涉及**钱、券、积分、实物、抽奖、会员权益**的功能或活动，上线前由业务负责人与风控同学共同逐项过。核心方法是**攻击者视角推演**：假设自己是职业羊毛党/黑产工作室，看这个规则怎么被榨干。每项写下推演结论与防控措施，留档备查。
>
> 配套使用：[活动方案模板](../模板/活动方案模板.md) 第 7 节、[活动上线前检查清单](活动上线前检查清单.md) 第四节。

---

## 一、激励漏洞场景推演

- [ ] **规则组合漏洞**：多个优惠是否可叠加（新人券+满减+会员价）导致实付为 0 或负毛利？叠加规则是否显式互斥
- [ ] **无限循环路径**：是否存在"参与→获奖→用奖励再参与"的正反馈循环，让单账号收益无上限
- [ ] **边界值攻击**：0.01 元下单、上限值±1、活动结束前 1 秒并发提交等边界场景的行为是否符合预期
- [ ] **并发与重放**：同一请求高并发重复提交（连点、脚本重放）会否重复发奖？发奖接口是否幂等
- [ ] **身份切换漏洞**：退登换号、卸载重装、换绑手机号后能否重复领取"一次性"奖励
- [ ] **取消回退漏洞**：下单领奖后取消订单/退款，奖励是否回收？优惠券用后退款，券是否退回并可再刷
- [ ] **客户端信任漏洞**：金额、次数、进度等关键数值是否有服务端校验，还是仅由客户端上报（客户端数据一律视为可伪造）

## 二、薅羊毛路径分析

- [ ] **批量账号成本测算**：注册一个可参与账号的黑产成本（手机号约 1-3 元）vs 单账号可套取收益——收益高于成本必被刷
- [ ] **设备维度限制**：单设备可参与账号数是否受限？模拟器、改机工具、群控设备能否被识别（接入设备指纹）
- [ ] **身份强度分级**：高价值奖励是否要求更强身份（实名/绑卡/历史消费），而非仅注册手机号
- [ ] **提现/变现路径卡点**：奖励变现（提现、转赠、兑换）处是否有二次风控？变现比参与更值得设卡
- [ ] **邀请裂变防互刷**：邀请奖励是否要求被邀请者完成有成本的动作（首单）而非仅注册？自邀、互刷、接码注册能否识别
- [ ] **地推/代理作弊**：有线下推广时，推广者自刷单的识别与追回机制是否存在
- [ ] **历史黑名单复用**：以往活动沉淀的作弊名单是否接入本次活动

## 三、预算熔断机制

- [ ] **多级熔断阈值**：小时级/日级/总预算三级熔断阈值已配置，且是系统自动执行而非"有人盯着"
- [ ] **发放速率监控**：奖励发放速率突刺（如 10 倍于均值）自动告警并可自动暂停发放
- [ ] **熔断后的用户体验**：熔断触发后页面降级文案（"今日名额已抢完"）已备好，不出现报错裸奔
- [ ] **熔断恢复流程**：谁评估、谁审批、如何恢复发放，流程与权限提前明确
- [ ] **资金对账**：发放流水与账务系统 T+1 对账，多发/异常发放能在 24 小时内发现

## 四、价格合规

- [ ] **划线价真实性**：原价/划线价有近期真实成交依据，不虚构原价再打折（价格欺诈红线）
- [ ] **先涨后降检查**：大促前是否存在提价再降价行为？促销价与近 30 天最低价的关系可解释
- [ ] **价格歧视合规**：差异化定价（新人价/会员价）规则是否公开透明；禁止基于用户画像的"大数据杀熟"式隐性差价
- [ ] **明码标价完整**：到手价构成（券后价、运费、附加费）在支付前清晰展示，无隐藏收费
- [ ] **广告法用语**：「全网最低」「史上最大优惠」等绝对化用语已清除

## 五、概率类玩法合规（抽奖/盲盒/开箱）

- [ ] **概率公示**：各奖项中奖概率在活动页可查（监管要求），公示概率与代码实际配置一致
- [ ] **概率真实性**：不存在"公示 1% 实际 0%"或"看人下菜"的动态概率（法律与舆情双重红线）
- [ ] **保底机制说明**：如有保底（N 次必中），规则清晰；如无保底，极端不中奖场景的客诉预案备好
- [ ] **未成年人保护**：涉及付费抽奖时，未成年人消费限制与退款通道符合监管要求
- [ ] **奖池库存兑现**：公示的大奖真实存在且可兑付，中奖名单可审计

## 六、退款与客诉预案

- [ ] **退款路径完整**：涉及付费的玩法有明确退款规则（何种情况可退、时效、原路退回），并在页面可查
- [ ] **奖励与退款联动**：退款时已发放的奖励/权益如何处理（回收/折抵），逻辑已实现并测试
- [ ] **误判申诉通道**：被风控拦截的正常用户有申诉入口，申诉处理时效有承诺（如 48 小时）
- [ ] **客诉分级响应**：普通咨询/资损投诉/媒体舆情三级响应机制与责任人明确
- [ ] **监管投诉预案**：12315/黑猫等平台投诉的接口人与标准处理流程明确
- [ ] **证据留存**：用户对规则的同意记录、概率配置变更记录、发放流水保留至少 2 年，以备纠纷举证

## 设计原则备忘（把风控做在规则里，而非事后拦截）

- **收益封顶**：任何玩法给单用户的总收益必须有上限，且上限 × 可批量的账号数 = 你的最坏损失——先算这笔账；
- **变现设卡**：黑产的目标是"变现"而非"参与"，把强校验放在提现/兑换处，对正常用户的参与体验影响最小；
- **有成本的动作才配奖励**：奖励挂在"完成首单""邀请者产生消费"这类有真实成本的动作上，而不是注册、点击这类零成本动作；
- **梯度信任**：新账号/新设备给低额度，随账号历史与消费行为积累逐步提额，代替一刀切；
- **留好熔断口**：所有发放逻辑必须可远程一键暂停，这是最后的安全网。

## 事故响应流程（发现被刷时）

1. **止血（分钟级）**：暂停发放/关闭提现入口，优先用预置开关而非发版；
2. **定损（小时级）**：拉发放流水，圈定异常账号集合与涉及金额，同步财务与法务；
3. **处置**：冻结未提现奖励；已流出资金评估追回可行性（司法/渠道协查）；处置话术统一，不透露识别逻辑；
4. **复盘**：按 [复盘报告模板](../模板/复盘报告模板.md) 归因到规则/风控/流程哪一层失守，防控措施回写本清单，形成累积。
