# 数据口径核对清单

> 使用说明：两份报表数字对不上、新指标上线、接手别人的报表、引用他人分析结论之前——先过一遍本清单。绝大多数"数据打架"不是数据错了，而是口径不一致。
>
> 用法：逐项核对并把结论写在指标文档或分析报告的口径附录里（参考 [分析报告模板](../模板/分析报告模板.md) 第 5 节、[指标字典模板](../表格/指标字典模板.md)）。排查数字对不上时，按本清单顺序二分定位，通常能在前 6 项内找到原因。

---

## 一、指标定义

- [ ] **指标名对应的定义一致**：同叫「活跃」，一方定义是"启动 App"，另一方是"有核心行为"——先对定义再对数字
- [ ] **分子分母事件明确**：转化率类指标的分子分母分别取哪个事件、哪个状态（如"支付"是发起支付还是支付成功）
- [ ] **与指标字典对齐**：口径与团队《指标字典》一致；不一致时，报告中显式声明差异及原因
- [ ] **单位与量纲**：金额是元还是分、含税否、人民币还是多币种折算（用哪天汇率）

## 二、统计周期

- [ ] **时间粒度一致**：自然日/自然周（周一起还是周日起）/自然月/滚动 7 天，双方是否相同
- [ ] **事件归属时间一致**：订单按下单时间、支付时间还是确收时间归属？留存按激活日还是注册日对齐？
- [ ] **数据就绪时间**：T+1 表在凌晨几点稳定？在数据未固化时间取数会出现"同一天两次取数不一样"
- [ ] **周期完整性**：对比的两个周期长度是否相等、是否都为完整周期（半个月 vs 整月不可比）

## 三、去重规则

- [ ] **去重键一致**：按 user_id、device_id、手机号还是账号去重？未登录流量如何计？
- [ ] **去重窗口一致**：日去重、周去重还是全周期去重？（周活 ≠ 7 个日活相加）
- [ ] **跨端合并规则**：同一用户在 App+H5+小程序多端活跃，算一个还是多个？ID 打通逻辑版本是否一致

## 四、过滤条件

- [ ] **内部与测试流量**：是否剔除员工账号、测试设备、预发环境数据
- [ ] **作弊与风控流量**：是否剔除风控标记用户？双方用的风控名单版本是否相同
- [ ] **业务范围过滤**：是否限定了渠道、地区、版本、业务线？对方是否也限定了
- [ ] **状态过滤**：订单类指标是否剔除取消单、退款单、0 元单、测试单

## 五、分母口径

- [ ] **分母人群一致**：留存率的分母是"新增设备"还是"新增注册"？渗透率的分母是 DAU 还是 MAU 还是全量注册用户
- [ ] **分母是否随时间变化**：漏斗各步分母是"上一步人数"还是"第一步人数"，两种算法结论完全不同
- [ ] **平均数的分母**：人均时长的分母是"全体用户"还是"使用过该功能的用户"，注意幸存者口径
- [ ] **比率的加权方式**：大盘率是"分子和/分母和"还是"各组率的简单平均"，样本不均时差异巨大（辛普森悖论入口）

## 六、时区

- [ ] **统计时区一致**：UTC、UTC+8 还是设备本地时区？跨国业务尤其注意
- [ ] **日切边界**：0 点边界的事件归哪天？服务端时间与客户端时间可能差数分钟到数小时
- [ ] **客户端时间可信度**：埋点若用客户端时间，需处理设备时间被改/穿越到未来的脏数据

## 七、埋点触发时机

- [ ] **曝光定义一致**：请求发出、开始渲染、渲染完成、还是可视面积 ≥50%？不同定义曝光量差数倍
- [ ] **上报机制差异**：实时上报还是批量延迟上报？延迟上报会导致当日数据次日仍在增长
- [ ] **端间实现一致**：同一事件在 iOS/Android/H5 的触发条件是否一致（常见：某端多埋/漏埋）
- [ ] **采样率**：埋点或日志是否采样？还原时乘回的采样系数是否正确
- [ ] **SDK 版本影响**：埋点 SDK 升级前后事件定义是否变化，变化点是否记录在案

## 八、历史数据可比性

- [ ] **口径变更点标注**：指标历史上何时改过口径？跨越变更点的同比环比必须注明或重刷
- [ ] **数据回刷记录**：历史数据是否被回刷过？报表快照与重跑结果不一致的原因是否可解释
- [ ] **业务事件标注**：大促、故障、政策变化、渠道结构突变等异常时段是否标注，避免误读趋势
- [ ] **产品形态变化**：功能入口改版、新端上线会造成指标"跳变"，与真实增长区分开

## 快速排查路径（数字对不上时）

1. 先对 **指标定义与过滤条件**（第一、四节）——约一半问题在这里；
2. 再对 **时间**（第二、六节）：粒度、归属、时区；
3. 再对 **去重与分母**（第三、五节）；
4. 仍不一致 → 各自取 10 条明细数据逐条比对，定位差异样本的共同特征；
5. 定位后，把结论回写进《指标字典》，让下一个人不再踩坑。

## 真实高频案例（对号入座）

- **"周活比日活加起来小很多"**：正常——周活按周去重，一人活跃 7 天只算 1 次；
- **"昨天的数今天又变了"**：延迟上报/回传归因导致 T+1 数据在 T+2 仍增长，取数应在数据固化后；
- **"两个 BI 看板同名指标差 3 倍"**：大概率一个含 H5 端一个不含，或一个剔除风控流量一个没剔；
- **"大盘涨了但每个渠道都在跌"**：辛普森悖论，高转化渠道占比上升所致，看结构而非只看大盘；
- **"活动 GMV 两个团队各报一个数"**：一方按下单口径、一方按支付口径，且对"活动 GMV"的归因范围（全站 vs 活动页引导）不同——立项时就该定义清楚。
